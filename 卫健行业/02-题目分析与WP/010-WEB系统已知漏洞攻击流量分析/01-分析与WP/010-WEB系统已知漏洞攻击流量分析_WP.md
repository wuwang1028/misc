# 010-WEB系统已知漏洞攻击流量分析 WP

## 一、题目信息

| 项目 | 内容 |
| --- | --- |
| 题号 | 010 |
| 题目名称 | WEB系统已知漏洞攻击流量分析 |
| 题型 | 附件分析题 |
| 提交格式 | `flag{xxxxxxx}` |
| 题目要求 | 攻击者似乎窥探到了 Web 服务器上的一个秘密，需要从攻击流量中恢复该秘密对应的 flag |

## 二、分析思路

本题附件为 HTTP 攻击流量包。初看流量中存在大量已知漏洞探测样本，包括 SQL 注入、文件读取、命令注入、ThinkPHP 利用、备份文件探测等，因此不能只停留在“攻击 IP 是谁”或“用了什么漏洞”这一层面，而要继续追踪**哪一次请求真正成功返回了敏感内容**。

对流量中的 HTTP 请求与响应做成对还原后，可以发现多数请求都返回了 `404 Not Found`，或者只是默认站点页面；真正有价值的成功响应集中在 `/atack?cmd=` 这一命令执行入口上。攻击者先通过命令执行验证目录结构，再进入日志目录，最终读取到 `secret.log` 的内容。

## 三、关键证据链

| 阶段 | 请求 | 关键响应 | 说明 |
| --- | --- | --- | --- |
| 命令执行验证 | `/atack?cmd=bHM=` | 返回 `apisix`、`logs` 等目录名 | 说明目标存在可利用的命令执行点 |
| 敏感文件定位 | `/atack?cmd=bHMgLWxhIC4vbG9ncw==` | 返回日志目录列表，出现 `secret.log` | 说明攻击者已在日志目录中发现敏感文件 |
| 秘密读取 | `/atack?cmd=Y2F0IC91c3IvbG9jYWwvYXBpc2l4L2xvZ3Mvc2VjcmV0LmxvZw==` | 返回 `JV4SA43FMNZGK5BANFZSAZTMMFTXWVDIGFZV6MLTL5WXSX3TGNRXEZLUL4ZDAMRTL5GDSTC7K4YW47I=` | 这是 `secret.log` 的实际内容 |

上述最后一条返回值不是随机噪声，而是一段 **Base32 编码字符串**。将其解码后得到：

```text
My secret is flag{Th1s_1s_my_s3cret_2023_L9L_W1n}
```

这说明题面所谓“被窥探到的秘密”就是明文中的 flag 值。

## 四、答案

```text
flag{Th1s_1s_my_s3cret_2023_L9L_W1n}
```

## 五、结论说明

本题最容易误判的地方在于：流量中充斥着大量已知漏洞扫描与失败探测，请求路径非常杂乱，且还夹杂一些明文口令、默认凭证和备份文件探测。如果只看攻击样本数量，很容易把目标误解为“找攻击 IP”或“找利用漏洞”。

但结合题面可知，题目核心是**找出攻击者最终看到的秘密**。因此必须继续追踪所有成功响应，并锁定真正返回敏感内容的那条命令执行链。最终只有 `cat secret.log` 这条请求直接给出了可解码的秘密内容，因此答案可以唯一确定为：`flag{Th1s_1s_my_s3cret_2023_L9L_W1n}`。
