# 主机基线加固 WP

## 一、题目概述

该题的目标是对目标主机执行一次**密码复杂度基线加固**。题面明确要求服务器密码策略应满足以下条件：密码长度至少 8 位，且至少包含 **1 个大写字母、1 个小写字母、2 个数字、1 个特殊符号**。题目同时说明 ttyd 的用户名与密码均为 `ctf`，并提示裁判机会定时检测是否完成加固，成功后生成 `/flag` 文件。

从题目性质来看，这并不是利用型题目，而是一个**运维基线落地题**。关键点不在于提权，而在于定位系统实际采用的认证与口令复杂度校验机制，再把策略准确修改到满足判题标准的粒度。

> 需要设置主机密码复杂度策略；要求至少 8 位，并包含 1 个大写字母、1 个小写字母、2 个数字、1 个特殊符号；裁判机会周期性检测，成功后生成 `/flag` 文件。

这说明本题无需提交 Web 表单或调用额外 API，真正的完成标志就是目标机本地出现 `/flag` 文件。

## 三、策略定位过程

进入系统后，我首先检查了 PAM 相关配置。结果显示，系统当前的关键密码模块配置位于 `/etc/pam.d/common-password`，且其中存在如下规则：

```text
password requisite pam_cracklib.so retry=3 minlen=8 difok=3
```

这一行表明系统已经使用 `pam_cracklib` 进行密码复杂度约束，但现有参数只覆盖了**最小长度**与**新旧密码差异**，并没有明确要求大写字母、小写字母、数字数量和特殊字符数量。因此，虽然配置“看起来像是已经做了加固”，但实际上并不满足题目给出的判题条件。

进一步检查文件权限后发现，`/etc/pam.d/common-password` 的权限为 `-rwxrwxrwx`。这意味着当前 `ctf` 用户可以直接修改该文件，而不需要 `sudo`。这正是题目预留的操作入口，也是本题的核心实施点。

## 四、修复思路

`pam_cracklib` 支持通过 credit 参数控制口令中不同字符类型的最小要求。为了满足题目条件，需要补充如下限制：

| 参数 | 含义 | 本题目标值 |
| --- | --- | --- |
| `minlen` | 最小密码长度 | `8` |
| `ucredit` | 至少需要的大写字母数量（负值表示最小需求） | `-1` |
| `lcredit` | 至少需要的小写字母数量（负值表示最小需求） | `-1` |
| `dcredit` | 至少需要的数字数量（负值表示最小需求） | `-2` |
| `ocredit` | 至少需要的其他字符数量，即特殊字符 | `-1` |

因此，正确的策略应该把原有行调整为：

```text
password requisite pam_cracklib.so retry=3 minlen=8 difok=3 ucredit=-1 lcredit=-1 dcredit=-2 ocredit=-1
```

这里保留原有的 `retry=3` 与 `difok=3`，只是在不破坏原始逻辑的前提下增加更细粒度的复杂度要求。这样既满足题意，也符合最小化修改原则。

## 五、实际修复步骤

为了保证操作可回滚，我先备份配置文件，再用一次性替换方式修改目标行。实际执行命令如下：

```bash
cp /etc/pam.d/common-password /tmp/common-password.bak
perl -0pi -e 's/pam_cracklib\.so\s+retry=3\s+minlen=8\s+difok=3/pam_cracklib.so retry=3 minlen=8 difok=3 ucredit=-1 lcredit=-1 dcredit=-2 ocredit=-1/' /etc/pam.d/common-password
```

修改完成后，再次核对关键配置，确认目标行已经变为：

```text
password requisite pam_cracklib.so retry=3 minlen=8 difok=3 ucredit=-1 lcredit=-1 dcredit=-2 ocredit=-1
```

此时无需手工改密码，也无需重启服务。根据题目 README 的说明，只要裁判机检测到当前策略满足要求，即会自动生成 `/flag`。

xctf{32b394377d7a44d69d98df63a59f78e6b89cb4d8}

## 八、题目本质总结

这题的本质是一个**面向 Linux 账户口令策略的最小改动型基线加固题**。它考察的不是复杂的渗透技巧，而是对 PAM 密码复杂度机制的理解、对 `pam_cracklib` 参数语义的掌握，以及在不具备 `sudo` 的表面限制下，识别出“可写配置文件”这一预留入口的能力。

若从复现角度看，题目的关键构成只有三部分：一是 Ubuntu 系统与 ttyd 登录环境，二是故意设置为弱化版的 `pam_cracklib` 配置，三是定时读取配置并在命中规则后写出 `/flag` 的判题脚本。因此它非常适合作为一个简洁的运维基线训练场景。
