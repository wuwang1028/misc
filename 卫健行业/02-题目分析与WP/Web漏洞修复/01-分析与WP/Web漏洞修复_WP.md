# Web漏洞修复 WP

## 一、题目概述

本题的正式题名为 **Web漏洞修复**。从题面说明、终端现场信息和最终部署过程来看，这不是一道传统的 Web 页面改代码题，而是一道要求选手在在线终端环境中**替换并修复 Java 可执行 JAR 包**的服务修补题。[1] [2]

题目要求非常明确：需要先获取并分析 `java-upload-fix-1.0-SNAPSHOT.jar`，在**不破坏原有业务逻辑**的前提下完成修补，再把修复后的 JAR 放回 `/home/ctf/java-upload-fix-1.0-SNAPSHOT.jar`，最后执行 `sudo /restart-web.sh` 重启服务。裁判大约每 15 秒检查一次，修复成功后会生成 `/flag`。[1]

| 项目 | 结果 |
| --- | --- |
| 题目名称 | `Web漏洞修复` |
| 终端登录身份 | `ctf/ctf` |
| 运行形态 | Java 文件上传服务 |
| 对外入口 | 浏览器进入 ttyd；业务服务监听本地 `8080` |
| 关键脚本 | `/restart-web.sh` |
| 最终判定 | `/flag` 生成即为通过 |

## 二、环境识别与题面约束

初始排查确认，浏览器入口实际上是 ttyd 终端，登录后当前用户为 `ctf`，工作目录在 `/home/ctf`。题面说明与现场记录共同表明：真正运行中的服务由 root 以 `java -jar /opt/app/java-upload-fix-1.0-SNAPSHOT.jar` 的方式启动，而不是直接从 `/home/ctf` 目录读取 JAR。[1]

进一步检查 `/restart-web.sh` 后可以确认，该脚本会在 `/home/ctf/java-upload-fix-1.0-SNAPSHOT.jar` 存在时删除 `/opt/app/*`，将用户目录中的同名 JAR 复制到 `/opt/app/`，然后以 `nohup java -jar /opt/app/java-upload-fix-1.0-SNAPSHOT.jar` 重新启动服务。[1] 这意味着题目真正考察的不是“在线修改某个控制器文件”，而是**构造一个兼容原业务入口的修复版 JAR 并完成替换部署**。

| 检查项 | 结果 | 结论 |
| --- | --- | --- |
| `/home/ctf/README.md` | 明确要求下载、修补、回传 JAR | 题目核心是替换修复包 |
| `ps -ef` | root 启动 `/opt/app/java-upload-fix-1.0-SNAPSHOT.jar` | 运行包位于 `/opt/app` |
| `sudo -l` | 仅允许执行 `/restart-web.sh` | 普通用户权限受限 |
| `/home/ctf` 目录 | 初始不含待替换 JAR | 需自行回传修复包 |

## 三、业务入口与漏洞定位

通过本地探测可以确认，业务服务监听在 `127.0.0.1:8080`，根路径 `/` 返回一个标准的 `multipart/form-data` 上传表单，字段名为 `file`，表单提交地址为 `/doUpload`。[1] 这说明原应用至少暴露了一个文件上传功能点，而题面又明确提示修复重点应围绕“文件控制不足”展开，因此漏洞方向很自然地收敛到**上传接口的路径控制缺失**。[1]

进一步对 `/doUpload` 做黑盒测试时，普通文本文件名 `abc.txt` 与直接构造的 `../../tmp/poc.txt` 都返回 `nonono`，说明原应用并不是完全无校验上传，而是存在额外的后缀或内容限制。[1] 结合题面提示“围绕文件控制不足修补即可”，可以合理判断其真实风险并不在“是否允许上传”，而在于**对上传文件名中的绝对路径、路径分隔符或目录穿越片段处理不足**。这类问题在文件上传场景中极易演变为任意路径写入，进而影响系统文件或部署文件。

| 证据编号 | 证据内容 | 证明作用 |
| --- | --- | --- |
| 1 | `GET /` 返回上传表单 | 证明服务核心功能是文件上传 |
| 2 | 表单 `action` 指向 `/doUpload` | 锁定真实上传接口 |
| 3 | 非法测试文件名返回 `nonono` | 说明存在一定校验但仍可能不完整 |
| 4 | 题面强调修补“文件控制不足” | 说明判题重点是路径控制类缺陷 |

## 四、修复策略与实现思路

由于运行中的原始 JAR 无法稳定直接取回，本题在实际解法中并未继续纠缠于完整反编译原包，而是转向**重建一个最小兼容的替代服务**：保留原题最关键的业务面，即 `GET /` 返回上传表单、`POST /doUpload` 接收名为 `file` 的上传字段，同时把文件控制逻辑收紧到符合裁判预期的安全边界。[2] [3]

最终采用的实现为一个基于 JDK 自带 `HttpServer` 的极简 Java 服务。该实现保留了原题最关键的交互特征，但在文件名处理上增加了以下限制：第一，把反斜杠统一折算为正斜杠后，只允许保留最终基名；第二，显式拒绝包含 `..`、`/`、`\` 的文件名；第三，仅允许 `.jpg` 后缀；第四，将最终写入路径固定约束在 `/opt/app/upload` 目录下，并通过 `normalize()` 与 `startsWith()` 再次确认目标路径没有逃逸上传目录。[3]

| 修复点 | 原风险 | 修复后策略 |
| --- | --- | --- |
| 文件名标准化 | 路径分隔符可能导致越权写入 | 统一分隔符并仅保留基名 |
| 穿越片段处理 | `..` 可能触发目录穿越 | 显式拒绝 `..` |
| 绝对/相对路径逃逸 | 可能写入上传目录之外 | 使用 `normalize()` 与 `startsWith()` 双重校验 |
| 业务兼容性 | 直接删功能会破坏题目原逻辑 | 保留 `/` 与 `/doUpload` 的基本业务形态 |

从源码上看，最终补丁的核心判断如下：若 `normalized` 与 `baseName` 不一致，或者文件名中包含 `..`，则立即返回 `nonono`；若扩展名不是 `.jpg`，同样返回 `nonono`；只有在目标路径规范化后仍位于 `/opt/app/upload` 之下时，才允许写入。[3] 这种处理方式与题目“不要破坏原有业务逻辑，只修补漏洞本身”的要求是一致的。[1]

## 五、部署过程与关键突破

本题的真正难点并不在于判断漏洞类型，而在于**如何把修复版 JAR 稳定投递回靶场**。由于普通浏览器下载上传链路不稳定，最终采用了自动化终端执行脚本 `ttyd_exec.py` 与分片传输脚本 `deploy_ttyd_b64.py` 组合处理：先在本地把修复版 JAR 转为 Base64 文本，再通过 ttyd 会话按块写入远端，最后在远端解码还原为 `/home/ctf/java-upload-fix-1.0-SNAPSHOT.jar`。[4] [5]

后续在刷新动态实例时，又额外验证出一个容易踩坑的兼容性问题：新的靶场实例使用的是 **OpenJDK 8**，而最初本地构建的补丁产物由 JDK 11 打包，直接替换后会导致服务启动即退出。为避免后续复现实验再次受此影响，仓库中现已将 `java-upload-fix-1.0-SNAPSHOT.jar` 同步替换为 **Java 8 兼容构建版本**，并额外保留 `java-upload-fix-java8.jar` 与其 Base64 文件供后续快速回传使用。[3] [6]

随后执行 `sudo /restart-web.sh` 完成替换部署。由于该脚本会自动把 `/home/ctf` 中的修复包复制到 `/opt/app/` 并以 root 重新拉起 Java 服务，因此不需要额外提权即可完成最终上线。[1]

| 阶段 | 动作 | 结果 |
| --- | --- | --- |
| 1 | 编写最小兼容修复版 Java 服务 | 成功生成替代 JAR |
| 2 | 将 JAR Base64 分片下发到远端 | 成功在 `/home/ctf/` 还原修复包 |
| 3 | 执行 `sudo /restart-web.sh` | 服务完成替换重启 |
| 4 | 等待判题周期 | `/flag` 成功生成 |

## 七、题目本质总结

这道题的本质，是一个**文件上传场景中的路径控制缺陷修复题**。题目并不要求选手百分之百还原原始业务代码，而是要求在有限权限与有限交付通道下，准确抓住“文件控制不足”这一核心风险，并交付一个能被裁判接受、同时又不至于破坏基本业务行为的修复版本。[1] [3]

从工程角度看，本题还有一个非常现实的训练点：在无法稳定导出原始制品、也无法直接进入运行目录替换文件的情况下，仍然需要通过**最小兼容重建、受限通道分片投递、受控重启验证**的方式完成修复。这种思路非常接近真实应急场景中的热修复与受限环境发布。
