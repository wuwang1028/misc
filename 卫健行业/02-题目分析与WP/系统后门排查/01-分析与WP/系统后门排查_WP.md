# 系统后门排查 WP

## 一、题目概述

该题属于 **Linux 主机后门排查与处置类题目**。题面给出的关键信息是：某台服务器的系统文件被攻击者篡改，导致**任意用户登录都会触发后门连接**；参赛者需要完成恶意内容清理，随后读取根目录下的 `flag` 并提交。

本题表面上看像是“提权取旗题”，但实际更接近**应急响应 + 持久化后门清理 + 判题触发**。真正的核心并不是传统漏洞利用，而是准确识别登录链路里的持久化点，完成最小必要修复，并让裁判脚本识别到“环境已恢复正常”，从而自动放出 `/flag`。

## 二、环境信息与判题特征

我进入题目环境后，首先确认了基本信息、运行进程与关键文件状态。综合排查结果，可以得到如下环境画像。

| 项目 | 结果 |
| --- | --- |
| 终端入口 | `http://oj-10-30-15-9-56797.adworld.xctf.org.cn/` |
| 终端类型 | `ttyd - Terminal` |
| 登录账号 | `ctf/ctf` |
| 当前用户 | `ctf` |
| 题目说明 | `~/README.md` 仅提示异常时可销毁容器重建 |
| 关键判题现象 | 处置完成后不会立刻提升当前权限，而是由裁判机异步生成 `/flag` |

进一步查看进程列表时，出现了一个非常关键的异常链路：

| PID | 用户 | 进程命令 | 含义 |
| --- | --- | --- | --- |
| `16` | `root` | `su ctf -c cd /tmp && while true ;do /var/www/html/application/model/.Backdoor.php ; sleep 10; done;` | root 启动一个切换到 `ctf` 的循环任务 |
| `17` | `root` | `python3 /root/check/judge.py` | 裁判脚本 |
| `18` | `root` | `/usr/sbin/ttyd -P 10 -p 18080 login` | Web 终端服务 |
| `19` | `ctf` | `sh -c cd /tmp && while true ;do /var/www/html/application/model/.Backdoor.php ; sleep 10; done;` | 实际持续执行恶意文件的循环进程 |

这说明题目并非单纯把后门放在某个孤立文件里，而是通过**系统登录链 + 持久化脚本 + 裁判检测**共同构成完整场景。

## 三、关键异常点定位

### 3.1 `/etc/profile` 被篡改

本题最核心的持久化点在 `/etc/profile`。该文件属于**所有交互式 shell 登录时都会加载的全局环境配置**，非常符合“任意用户登录都会触发”的题面描述。

排查时发现：

| 检查项 | 结果 |
| --- | --- |
| 文件路径 | `/etc/profile` |
| 权限 | `-rwxrwxrwx` |
| 风险级别 | 极高 |
| 风险原因 | 被任何普通用户写入，且登录即触发 |

这一步其实已经基本锁定题目方向。因为只要攻击者把恶意命令植入 `/etc/profile`，无论是通过 ttyd 登录，还是通过其他交互式 shell 登录，都会自动执行后门逻辑。

### 3.2 Web 目录下存在隐藏恶意文件 `.Backdoor.php`

继续结合进程命令行可知，循环执行的目标文件为：

```text
/var/www/html/application/model/.Backdoor.php
```

该文件的几个特征都非常典型：

| 特征 | 表现 |
| --- | --- |
| 文件名 | 以点开头，属于隐藏文件 |
| 所在目录 | 位于 Web 应用模型目录下，具备迷惑性 |
| 权限 | 对 `ctf` 可控 |
| 作用 | 被循环调用，形成持续性后门触发 |

题目环境中还能看到其备份副本 `.Backdoor.php.bak`，说明在清理过程中对原始恶意文件做了隔离留存，便于后续复盘。

### 3.3 恶意逻辑并非一次性触发，而是循环执行

进程 `16/19` 表明，题目环境里并不是“登录一次触发一次然后结束”，而是通过一个 `while true` 循环让恶意文件持续执行。这个设计有两个目的：

第一，它强化了“后门仍在活动中”的排查感受；第二，它使得**仅删除文件但不处理运行中的恶意进程时，环境未必会立刻恢复到裁判认可状态**。

因此，本题的真正处置应当分成三步：

| 步骤 | 目标 |
| --- | --- |
| 第一步 | 从登录链路中移除触发点，即修复 `/etc/profile` |
| 第二步 | 删除或隔离恶意文件 `.Backdoor.php` |
| 第三步 | 终止仍在运行的恶意循环进程，避免旧会话残留影响判题 |

## 四、处置思路

本题采用**最小改动原则**进行修复，即尽量不破坏环境其它部分，只清除题目明确要求的恶意内容。

整体思路如下。

| 阶段 | 操作目标 | 说明 |
| --- | --- | --- |
| 取证备份 | 备份 `/etc/profile` | 保留原始证据，防止误操作后无法回溯 |
| 配置修复 | 删除 `/etc/profile` 中指向 `.Backdoor.php` 的恶意启动项 | 解除“任意用户登录即触发” |
| 文件处置 | 将 `.Backdoor.php` 改名为 `.Backdoor.php.bak` | 避免再次执行，同时保留样本 |
| 进程处置 | 杀死 `ctf` 侧循环进程 `19` | 使 root 侧 `su` 进程一并退出 |
| 判题等待 | 重新建立新会话并检查 `/flag` | 让裁判脚本确认环境已恢复 |

## 五、实际操作过程

### 5.1 备份与修复 `/etc/profile`

为了避免直接覆盖造成不可逆错误，先备份原始文件，再去掉与 `.Backdoor.php` 相关的恶意行。实际思路可以概括为：

```bash
cp /etc/profile /tmp/profile.bak_ctf
grep -v 'Backdoor.php' /etc/profile > /tmp/profile.clean
cat /tmp/profile.clean > /etc/profile
```

执行后再次核查，`/etc/profile` 中已不再包含 `Backdoor.php` 相关内容，可认为**全局登录触发点已被清除**。

### 5.2 隔离恶意文件 `.Backdoor.php`

在登录链修复之后，还需要把实际后门文件移走，否则旧循环进程仍可能继续尝试调用它。处理方式如下：

```bash
mv /var/www/html/application/model/.Backdoor.php \
   /var/www/html/application/model/.Backdoor.php.bak
```

验证结果表明，原始恶意文件已消失，仅剩备份样本：

| 文件 | 状态 |
| --- | --- |
| `/var/www/html/application/model/.Backdoor.php` | 不存在 |
| `/var/www/html/application/model/.Backdoor.php.bak` | 存在 |

### 5.3 终止持续运行的后门循环进程

清理完文件后，进程列表里仍能看到此前由 root 启动、由 `ctf` 持续执行的循环任务。为了让环境恢复到稳定状态，需要结束这条进程链。

关键操作为：

```bash
kill -9 19
```

处理前后的进程对比如下。

| 阶段 | 现象 |
| --- | --- |
| 处理前 | 存在 `root -> su ctf -> while true -> .Backdoor.php` 的完整链条 |
| 杀死 `19` 后 | `ctf` 侧循环消失，root 侧对应 `su` 进程也一并退出 |

这一步是本题是否能成功放出 flag 的关键补充动作。仅修复文件不处理遗留进程，裁判未必会立即认定环境已经恢复。

## 六、flag 获取过程

在完成以上三步后，我重新建立了一个新的 ttyd 登录会话，再次检查根目录，发现题目环境已经生成可读的 `/flag` 文件：

| 路径 | 权限 | 状态 |
| --- | --- | --- |
| `/flag` | `-rwxrwxrwx` | 已生成，可直接读取 |
| `/readflag` | `-r-x------` | 仍不可由 `ctf` 直接执行 |
| `/etc/fec956a165add50fe621cdabc9c31d4c` | `-r--------` | 仍不可直接读取 |

这说明本题并不是要求参赛者真正完成 root 提权，而是要求**通过清理后门触发裁判逻辑，由裁判主动把 flag 落地到根目录**。

最终读取 `/flag`，得到结果：

```text
xctf{ea9a4b4914d944888691223d2b4cb133227a07e7}
```

| 证据编号 | 证据内容 | 证明作用 |

## 八、题目本质总结

这道题本质上考察的是**Linux 登录持久化后门排查**，而不是传统意义上的漏洞利用。它要求参赛者把题面语言“任意用户登录触发”准确映射到 `/etc/profile` 这类全局登录配置，再结合运行中的恶意循环进程，还原出攻击者的持久化方式。

从训练视角看，这题主要覆盖了以下能力：

| 能力点 | 说明 |
| --- | --- |
| 系统级持久化识别 | 能从题面特征联想到 `/etc/profile` |
| 恶意文件定位 | 能从进程命令行回溯到隐藏样本 `.Backdoor.php` |
| 运行态处置 | 认识到清理文件后还要终止遗留进程 |
| 判题理解 | 明确题目真正的过关条件是“裁判生成 `/flag`” |

因此，本题最准确的通关口径应当表述为：**清理全局登录持久化项、隔离恶意脚本、终止残留循环进程，等待裁判机生成 `/flag`，而不是强行提权读取 root 私有文件**。
