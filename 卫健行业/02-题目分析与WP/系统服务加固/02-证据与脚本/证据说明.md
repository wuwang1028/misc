# 证据说明

本目录用于存放《系统服务加固》一题在分析、修复、验证与取旗过程中形成的关键证据文件，便于后续审阅 WP、复核结论以及在其他环境中复现实验链路。

## 一、关键结论

本题最终通过条件并不是单独完成 FTP 加固，而是同时满足以下两项：

1. **Redis 已启用口令认证**，错误口令认证结果应返回 `WRONGPASS`；
2. **FTP 匿名登录已被禁止**，即 `anonymous_enable=NO` 生效。

在上述两个条件同时成立后，裁判约每 15 秒检查一次，并最终生成 `/flag`。

## 二、文件清单

| 文件名 | 作用 | 关键内容 |
| --- | --- | --- |
| `system_service_hardening_live_findings.md` | 第二题现场记录 | 汇总题面、进程、配置权限、真实判题逻辑、最终修复动作与 flag |
| `second_challenge_judge_and_flag.out` | 核心终端回显证据 | 包含导出的 `judge.py` 内容、最终 FTP 关键配置状态、`/flag` 存在与 flag 输出 |
| `ttyd_exec.py` | 终端自动化脚本 | 用于稳定登录 ttyd 环境并批量执行命令，是本题取证与修复的重要辅助脚本 |

## 三、核心证据链说明

首先，`system_service_hardening_live_findings.md` 记录了本题从“误以为只需处理 FTP”到“确认 Redis 也进入裁判逻辑”的完整演进过程。该文件明确说明，单做 FTP 配置调整不足以出 flag，真正的裁判条件是 Redis 认证与 FTP 匿名访问限制同时成立。

其次，`second_challenge_judge_and_flag.out` 是最关键的直接证据。该文件中可以看到导出的 `/root/check/judge.py` 内容，裁判脚本先对 `127.0.0.1:6379` 发送错误 `AUTH`，只有当返回 `WRONGPASS` 时才继续检查匿名 FTP 登录是否成功。文件尾部还能直接看到 `/flag` 已生成以及最终 flag 的具体内容。

最后，`ttyd_exec.py` 说明本题并非纯粹手工完成，而是在受限终端环境中结合自动化执行完成稳定取证、配置调整和验证。这也是后续环境复现时最值得保留的辅助材料之一。

## 四、最终取旗依据

本题最终通过的直接依据如下：

- Redis 错误口令认证已返回 `WRONGPASS`；
- FTP 匿名访问已关闭；
- `/flag` 文件存在；
- `/flag` 内容为：

```text
xctf{0c8ad26cb1fb4cc4855b73dce8739c8a81fe78f4}
```
