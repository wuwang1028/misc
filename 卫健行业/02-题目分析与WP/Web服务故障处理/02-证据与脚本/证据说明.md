# Web服务故障处理证据说明

## 一、证据用途说明

本目录用于保存《Web服务故障处理》题目的关键排障输出、自动化执行脚本以及最终修复验证结果。由于本题不是单纯依靠一条命令即可完成，而是需要逐步确认源码逻辑、接口路径、环境变量可见性、PHP-FPM 行为差异和最终放旗条件，因此保留完整证据链对于后续复盘、WP 撰写和复现环境设计都很重要。

## 二、核心结论

本题的最终结论可以概括为以下三点。第一，应用源码中的数据库密码已经改为通过 `getenv('DATABASE_PASSWORD')` 读取。第二，系统 Shell 环境和 PHP CLI 中都能读取到该环境变量，但 Web 请求上下文中的 PHP-FPM 无法读取，因此数据库连接失败。第三，在当前普通用户权限下无法直接修改 FPM 池配置，于是通过把真实密码 `Passw0rd@123456789` 直接写回 `application/database.php` 恢复业务，随后触发裁判生成 `/flag`。

## 四、关键验证过程摘要

为了让后续复盘更高效，可以把本题的关键验证过程整理成如下顺序。

| 阶段 | 动作 | 结论 |
| --- | --- | --- |
| 1 | 检查 Web 与 ThinkPHP 控制器 | 应用功能点存在，但数据库相关接口报错 |
| 2 | 读取 `database.php` | 密码配置依赖 `getenv('DATABASE_PASSWORD')` |
| 3 | 在 Shell 中查看环境变量 | `DATABASE_PASSWORD=Passw0rd@123456789` 存在 |
| 4 | 用 PHP CLI 测试 `getenv()` | CLI 可以读到密码 |
| 5 | 在 Web 根目录放置 `envtest.php` 并访问 | Web/FPM 上下文返回 `bool(false)` |
| 6 | 检查 `/usr/local/etc/php-fpm.d/www.conf` | `clear_env` 未生效，导致环境变量不透传 |
| 7 | 尝试直接修复 FPM 配置 | 因当前用户权限不足而失败 |
| 8 | 采用应急修复写回真实密码 | `Check` 与 `getuser` 接口恢复 |
| 9 | 等待判题 | 根目录生成 `/flag` |

## 五、最终取旗结果

最终通过如下思路完成取旗：

```bash
sed -i "s/getenv('DATABASE_PASSWORD')/'Passw0rd@123456789'/" /var/www/html/application/database.php
curl -sS http://127.0.0.1/public/index.php/index/check/testmysql
curl -sS http://127.0.0.1/public/index.php/index/index/getuser
cat /flag
```

最终 flag 为：

```text
xctf{d58e5840d29b49c497412467a5f6539b8abbb8f9}
```

## 六、归档说明

本题后续若继续补造成完整复现环境，最值得保留的不是单个脚本，而是以下完整链条：题面提示数据库密码在环境变量中；应用配置使用 `getenv()`；Shell 与 CLI 能读到密码；Web/FPM 请求上下文读不到密码；因此数据库连接失败；恢复配置或运行环境后，`Check.php::testmysql()` 可成功返回固定 MD5；最终触发 `/flag` 生成。只要这条因果链保持不变，题目的训练价值就能完整复用。
