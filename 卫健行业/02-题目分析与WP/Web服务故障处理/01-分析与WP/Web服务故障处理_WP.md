# Web服务故障处理 WP

## 一、题目概述

本题的正式题名为 **Web服务故障处理**。根据题面描述可知，某公司在上一次被黑客入侵数据库后，虽然及时修改了数据库密码，但却因此导致 Web 服务出现故障。题面同时明确给出了一个关键提示：**数据库密码位于环境变量中**。此外，登录方式不是传统 SSH，而是通过浏览器访问在线终端，终端账号和密码均为 `ctf`。

从题面语义来看，这道题并不是让选手继续利用漏洞攻击数据库，而是要求选手站在应急响应和故障处置的角度，恢复一个因为“错误修复”而失效的 Web 应用。也就是说，核心任务不是找漏洞，而是要识别出应用、运行环境与数据库凭据之间的真实耦合关系，修复服务，使裁判访问点恢复正常，随后等待系统放旗。

## 二、题面识别与环境信息

结合题面截图、实例页面以及登录后环境信息，可以先对这道题做一个基础画像。

| 项目 | 结果 |
| --- | --- |
| 题目名称 | `Web服务故障处理` |
| 入口形式 | Web 页面 + 在线终端 |
| 终端登录账号 | `ctf/ctf` |
| 服务类型 | Nginx + PHP-FPM + ThinkPHP 5 |
| 题面核心线索 | 数据库密码位于环境变量中 |
| 排查目标 | 修复 Web 服务与数据库连接故障 |
| 最终取旗方式 | 服务恢复并满足裁判条件后生成 `/flag` |

题面中的“数据库密码位于环境变量中”是本题最重要的一条提示。它意味着故障很可能并不是数据库本身不可用，而是应用对密码的读取方式、运行环境的环境变量传递机制，或者修复时把密码写错了地方。换言之，排查重点应该优先落在 **应用配置、环境变量可见性、PHP-FPM 运行机制** 这三个层面，而不是盲目去改数据库权限或重装服务。

## 三、初始排查与异常定位

登录在线终端后，首先对系统、服务和应用进行首轮扫描。排查结果表明，Web 服务相关组件仍然在运行，但业务接口无法正常返回数据库查询结果，说明问题更像是“服务能启动、应用能路由、但连库失败”。

| 检查项 | 结果 | 结论 |
| --- | --- | --- |
| 当前用户 | `ctf` | 只有普通用户权限 |
| Web 服务进程 | 存在 | 站点基础服务并未整体宕掉 |
| 框架类型 | ThinkPHP 5.0.24 | 可以按 ThinkPHP 目录结构继续排查 |
| 首页访问 | 可打开框架默认页或业务入口 | 说明 Nginx/PHP-FPM 基本通路仍在 |
| 数据查询接口 | 返回错误页 | 重点怀疑数据库连接配置异常 |

随后继续检查应用源码中的控制器与判题相关文件。`Index.php` 中的 `getuser()` 方法会调用 `Db::name('user')->find()` 查询数据库，并在成功时输出用户信息；`Check.php` 中的 `testmysql()` 方法同样会执行数据库查询，若成功则回显固定字符串的 MD5 值。对应逻辑如下。

| 文件 | 关键函数 | 作用 |
| --- | --- | --- |
| `application/index/controller/Index.php` | `getuser()` | 查询 `user` 表并输出 `id/username/password` |
| `application/index/controller/Check.php` | `testmysql()` | 成功连库后输出固定 MD5，供裁判判定 |

这两个控制器给出了非常明确的判断信号：**只要数据库连接恢复正常，`getuser()` 和 `testmysql()` 都应能够成功返回内容**。因此，本题的修复目标不需要猜测，而是可以直接用接口回显作为验证标准。

## 四、关键证据链分析

在排查数据库配置文件 `application/database.php` 时，可以看到密码字段并不是写死密码，而是采用了环境变量方式读取：

```php
'password' => getenv('DATABASE_PASSWORD'),
```

这与题面“数据库密码位于环境变量中”的描述表面上是一致的。但进一步验证后发现，问题并没有这么简单。命令行环境中执行 `env | grep DATABASE_PASSWORD` 能看到：

```text
DATABASE_PASSWORD=Passw0rd@123456789
```

同时，在命令行直接执行 PHP 测试代码时，`php -r 'var_dump(getenv("DATABASE_PASSWORD"));'` 也能正确输出密码字符串。这说明 **系统层面的环境变量本身是存在的**。然而，当把一个临时测试文件放入 Web 根目录并通过浏览器访问时，返回结果却是：

```php
bool(false)
```

这一步非常关键。它说明同样的 `getenv('DATABASE_PASSWORD')`，在 **CLI 上下文** 中可以读到值，但在 **PHP-FPM 处理 Web 请求的上下文** 中却读不到。这也就解释了为什么源码看起来“写法正确”，实际业务仍然报错。

为了进一步确认这一点，又检查了 PHP-FPM 的池配置文件 `/usr/local/etc/php-fpm.d/www.conf`。其中可以看到：

```text
;clear_env = no
```

这行配置处于注释状态。在 PHP-FPM 的默认行为下，`clear_env` 未关闭时，子进程处理 Web 请求时不会继承宿主环境中的大部分变量。因此，虽然题目环境里确实设置了 `DATABASE_PASSWORD`，但 Web 应用运行时的 `getenv()` 依然返回空值，最终导致 ThinkPHP 在连接数据库时失败。

由此，本题的真实故障链条就完整了。

| 证据编号 | 证据内容 | 证明作用 |
| --- | --- | --- |
| 1 | 题面说明数据库密码位于环境变量中 | 明确排查重点是环境变量机制 |
| 2 | `database.php` 使用 `getenv('DATABASE_PASSWORD')` | 证明应用依赖环境变量取密 |
| 3 | Shell 中 `DATABASE_PASSWORD` 存在 | 说明变量并未丢失 |
| 4 | CLI 执行 `getenv()` 成功 | 说明 PHP 本身支持该取值方式 |
| 5 | Web 访问 `envtest.php` 返回 `bool(false)` | 证明 PHP-FPM 的 Web 上下文无法继承该变量 |
| 6 | `www.conf` 中 `;clear_env = no` 仍被注释 | 说明 FPM 环境变量未放开 |
| 7 | `Check.php::testmysql()` 依赖数据库查询成功 | 说明修复后应以该接口回显作为判题信号 |

## 五、故障根因判断

基于上述证据，可以得出本题的真正根因：

> **应用配置文件已经改成从环境变量读取数据库密码，但 PHP-FPM 默认清除了运行环境变量，导致 Web 请求上下文中的 `getenv('DATABASE_PASSWORD')` 取值失败，最终引发数据库连接错误。**

这也是本题最有迷惑性的地方。因为从代码层面看，开发者似乎已经按题面要求把数据库密码改成了环境变量读取；但从运行时角度看，**“代码写法正确”不等于“部署方式正确”**。如果 Web 服务容器中的 FPM 不透传环境变量，那么应用代码依旧会拿到空密码。

从真实运维场景看，这类问题非常典型。很多系统在把敏感信息从配置文件迁移到环境变量后，都会忽视中间运行层对环境变量的清洗策略，尤其是在 Nginx + PHP-FPM、Supervisor、Systemd 等多层组件叠加时，这种“代码没问题、运行却失败”的现象尤其常见。

## 六、修复思路与实际处置

理论上，这道题最完整、最规范的修复方式应当是修改 PHP-FPM 池配置，把 `clear_env` 关闭，并显式透传 `DATABASE_PASSWORD`，例如：

```ini
clear_env = no
env[DATABASE_PASSWORD] = $DATABASE_PASSWORD
```

随后重载 PHP-FPM，使 Web 请求上下文能够真正读取到环境变量。

不过，在本次靶场实例中，当前登录用户 `ctf` 无法直接改写 `/usr/local/etc/php-fpm.d/www.conf`，尝试修复时会遇到权限拒绝。因此，在现有权限边界下，需要采用**可落地的等效修复方案**：把已经确认存在于环境变量中的真实数据库密码直接写回应用配置文件，使业务先恢复正常。

实际修复命令如下：

```bash
sed -i "s/getenv('DATABASE_PASSWORD')/'Passw0rd@123456789'/" /var/www/html/application/database.php
```

修复后的关键验证结果如下。

| 验证项 | 修复后结果 | 说明 |
| --- | --- | --- |
| `database.php` 密码字段 | `'Passw0rd@123456789'` | 应用已能直接使用真实密码 |
| `Check` 接口 | 返回 `8e74933d7c6fa65e1aab5769743447ed` | 裁判检测点恢复正常 |
| `getuser` 接口 | 输出用户信息 | 数据库查询链路已恢复 |
| `/flag` | 成功生成 | 题目已判定修复成功 |

其中，`getuser()` 接口修复后返回了如下业务数据：

```text
id: 1
username: admin
password: 123456
```

而 `Check.php::testmysql()` 修复后返回的校验值为：

```text
8e74933d7c6fa65e1aab5769743447ed
```

这表明裁判访问点已经能够成功完成数据库查询，因此服务故障已经被修复到裁判认可的状态。

## 八、题目本质总结

这道题本质上考察的是 **Web 应用配置修复能力**，而不是 Web 漏洞利用能力。它把题面线索、源码阅读、运行环境机制与裁判逻辑组合成了一个完整闭环：题面告诉我们密码在环境变量里；源码表明应用依赖 `getenv()`；运行结果又提示 Web 请求上下文读取失败；最终需要选手把“代码配置”与“运行机制”两者统一起来，才能真正恢复服务。

从训练价值看，这道题至少覆盖了三个非常实用的知识点。第一，选手要能够区分 **CLI 与 FPM/Web 请求上下文** 的环境差异。第二，选手需要理解 `getenv()`、PHP-FPM `clear_env`、池配置透传变量之间的关系。第三，选手还要意识到 CTF 中很多“服务故障处理”题的通关标准不是抽象地“修好了”，而是要精确恢复到 **裁判接口可检测** 的状态。

也正因为如此，本题最准确的解法总结应当是：**定位到 ThinkPHP 应用依赖环境变量读取数据库密码，但 PHP-FPM 的 Web 上下文无法继承该变量，导致数据库连接失败；在当前权限下通过把真实密码写回配置文件恢复业务，使判题接口 `Check::testmysql()` 成功返回校验值，进而触发 `/flag` 生成。**
