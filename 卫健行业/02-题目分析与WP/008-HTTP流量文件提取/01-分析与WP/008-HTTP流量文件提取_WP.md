# 008-HTTP流量文件提取 WP

## 一、题目信息与最终结论

本题要求根据给出的监测流量文本，找出被窃取的是哪个数据库，提交格式为 **`flag{数据库名称}`**。对附件中的 `wenjiantiqu.pcapng` 进行 HTTP 会话解析、二进制字符串检索与对象内容定位后，可以明确恢复出一个通过 HTTP 响应传输的 ZIP 压缩对象，其内部数据库导出文件名为 **`wstmart.sql`**。因此，本题对应的数据库名称为 **`wstmart`**，最终提交内容为：

```text
flag{wstmart}
```

本题的关键在于，常规 HTTP 头部里并没有直接把数据库名写成明文字段，因此不能停留在表面请求路径或响应头分析，而必须进一步恢复响应体中的压缩对象内容，才能拿到真正的数据库名称。

## 二、分析思路

流量题中“被窃取的是哪个数据库”通常对应的是导出文件名、压缩包内文件名或者数据库备份脚本中的 `CREATE DATABASE` 等标识。为避免遗漏，我先对流量包做程序化解析，提取 HTTP 请求与响应；随后又对原始流量做二进制可读检索，重点搜索 `.sql`、数据库名、压缩文件头 `PK` 等线索。

这种双路径分析很有必要。因为在实际流量中，HTTP 响应体往往并不会被完整打印成可读文本，但压缩包文件头、文件名字符串以及部分对象元数据仍然会以明文形式残留在重组后的对象内容中。最终，也正是这一层证据给出了明确答案。

## 三、关键证据

在流量二进制内容中，可以恢复出一个通过 HTTP 下载返回的 ZIP 压缩对象。该对象具备典型的 ZIP 文件头，并且内部文件名清晰可见为 **`wstmart.sql`**。

| 证据类型 | 关键内容 | 说明 |
| --- | --- | --- |
| HTTP 响应对象 | `Content-Type: application/zip` | 返回的是压缩包下载对象 |
| 压缩包内容痕迹 | `PK...wstmart.sql...` | ZIP 内含数据库导出文件 |
| 文件扩展名 | `.sql` | 明确属于数据库导出结果 |
| 去除扩展名后的库名 | `wstmart` | 应作为最终提交值 |

阶段性分析中已经确认如下结论：

```text
HTTP 响应体中可恢复 ZIP 对象，压缩包内部文件名为 wstmart.sql
```

这条证据足以说明攻击者通过 HTTP 传输的实际目标并不是任意文档，而是一个数据库导出文件。题目问的是“哪个数据库”，因此应取文件名去掉 `.sql` 后的数据库名，而不是提交完整文件名。

## 四、排除性判断

在常规 HTTP 头部、请求路径和普通参数中，并没有直接出现更明确的数据库名称候选。因此，如果只看请求行或普通响应头，很容易得不到答案，或者误把文件名、压缩包名、临时路径当成数据库名。

相比之下，`wstmart.sql` 具备三个决定性特征。第一，它明确出现在被传输的压缩对象内部；第二，它带有标准数据库导出文件扩展名 `.sql`；第三，去掉扩展名后能够形成自然且完整的数据库名称 `wstmart`。因此该结论比其他零散字符串更稳定、更符合题面要求。

## 五、最终答案

综合 HTTP 对象恢复结果与数据库导出文件名判断，本题最终提交为：

```text
flag{wstmart}
```
