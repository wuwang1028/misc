# 001-系统日志联合分析 WP

## 一、题目信息与最终结论

本题要求基于给出的 SSH 日志，识别“可疑的入侵者”来源 IP。结合题目截图恢复结果，提交格式为 **`flag(IP地址)`**。对全部认证日志进行时间线梳理后，可以确认最符合入侵特征的地址为 **`177.77.135.158`**，对应最终提交内容为：

```text
flag(177.77.135.158)
```

从日志行为上看，这个地址并不是普通的单次登录来源，而是先对 `root` 账户执行短时间连续口令尝试，随后紧接着完成成功登录，呈现出十分典型的**暴力破解后登录成功**特征。因此，它明显比其他零散的成功登录地址更符合题面中“可疑入侵者”的定义。

## 二、分析思路

本题的核心不是简单寻找“出现过成功登录的 IP”，而是要从 SSH 认证日志里区分**正常运维访问**与**异常攻击行为**。因此分析时需要同时关注失败认证、认证次数异常、认证限制触发以及最终成功登录这几类事件，并观察它们是否在同一时间窗口内串成完整链条。

为了避免误判，我首先对认证日志中的成功登录来源做了归纳，再对其中每个地址向前回溯失败记录。经过对比，只有 `177.77.135.158` 同时具备“多次失败”“达到认证次数上限”以及“紧接着成功登录”三个关键特征，因此可以作为最终答案。

## 三、关键证据

下表展示了最关键的攻击链条。

| 时间 | 日志行为 | 关键含义 |
| --- | --- | --- |
| `Oct 14 17:52:08` | `Failed password for root from 177.77.135.158` | 开始对 `root` 账户尝试口令 |
| `Oct 14 17:52:25` | `message repeated 5 times: [ Failed password for root from 177.77.135.158 ... ]` | 短时间重复失败，具备爆破特征 |
| `Oct 14 17:52:26` | `maximum authentication attempts exceeded for root from 177.77.135.158` | 触发认证尝试次数限制 |
| `Oct 14 17:52:35` | `Accepted password for root from 177.77.135.158 port 51948 ssh2` | 爆破后成功登录 |

对应的证据片段如下。

```text
Oct 14 17:52:08 ... Failed password for root from 177.77.135.158 ...
Oct 14 17:52:25 ... message repeated 5 times: [ Failed password for root from 177.77.135.158 ... ]
Oct 14 17:52:26 ... maximum authentication attempts exceeded for root from 177.77.135.158 ...
Oct 14 17:52:35 ... Accepted password for root from 177.77.135.158 port 51948 ssh2
```

这组日志最有价值的地方在于，它不是孤立的一条成功认证，而是完整展现了攻击者从失败尝试到最终得手的全过程。相比之下，其他出现过成功登录的 `100.*` 地址更像历史管理访问或常规登录来源，因为没有在其前后看到同样强烈的异常试错行为。

## 四、排除性判断

为了避免把普通运维地址误判为攻击者，我进一步从行为模式上做了区分。

| 地址类型 | 观察结果 | 判断 |
| --- | --- | --- |
| `177.77.135.158` | 失败尝试密集、触发认证限制、随后成功登录 | 高度可疑，符合入侵特征 |
| 其他零散成功地址 | 能看到成功登录，但未表现出连续爆破与异常认证特征 | 更像正常管理访问或历史登录 |

这说明本题不能只看“谁登录成功过”，而必须看**该登录是否由异常试探行为铺垫而成**。`177.77.135.158` 是当前证据里唯一满足这一条件的地址，因此结论稳定。

## 五、最终答案

综合题面要求与日志证据，本题最终提交为：

```text
flag(177.77.135.158)
```
