# 030-加密Webshell日志分析 WP

## 一、题目信息

| 项目 | 内容 |
| --- | --- |
| 题号 | 030 |
| 题目名称 | 加密Webshell日志分析 |
| 题型 | 附件分析题 |
| 提交格式 | `flag{xxx}` |
| 题目目标 | 根据访问日志找出攻击者真实攻击载荷密码 |

## 二、分析思路

本题附件为 `access.log`，内容是结构化 JSON 访问日志。日志中前半段主要是大规模漏洞扫描与通用利用探测，绝大多数请求均返回 `404` 或 `302`，不具备真实落地价值。要找到题面所说的“真实攻击载荷密码”，关键在于从海量噪声中识别出**真正成功、并被持续利用的 Webshell 入口**。

统计后可以发现两个显著的可疑入口：`/font.php` 与 `/lang.php`。其中 `/lang.php` 在 17:06 左右出现大量返回 `200` 的 POST 请求，请求体整体表现为 Base64 风格密文，更像 Behinder 或 Godzilla 一类加密管理流量；但单纯从访问日志并不能直接恢复口令。相比之下，`/font.php` 的请求体明确包含一段固定参数名加上一整段 PHP 载荷，并且该参数名在多条成功请求中稳定复用，因此更符合题面要找的**真实攻击载荷密码**。

## 三、关键证据链

### 1. 成功攻击入口锁定为 `/font.php`

在访问日志中，`/font.php` 存在多条连续的 `200` 响应 POST 请求。例如：

```text
POST /font.php HTTP/1.1
status=200
```

请求体开头反复出现以下固定结构：

```text
5wxBnp5iX879uyP1d2js3MIOVrYFS76U=...
```

这一串位于等号左侧，后接 URL 编码的 PHP 代码，明显是单行 Webshell 的**入口参数名**，也就是攻击端实际使用的口令。

### 2. 固定参数名在真实攻击中持续复用

日志显示，攻击者在 `/font.php` 上执行了多种不同功能：

| 时间段 | 动作特征 | 说明 |
| --- | --- | --- |
| 17:00 左右 | 目录遍历、文件列表 | 使用动态字段 `m9ad948a8dfa85` 等传递功能参数 |
| 17:00 至 17:04 | 命令执行、环境探测 | 使用 `te132822326775`、`x75801a1a4fa8d` 等字段传递命令与解释器 |
| 全部阶段 | 固定开头均为 `5wxBnp5iX879uyP1d2js3MIOVrYFS76U=` | 说明这是稳定入口密码，而非一次性内部变量 |

如果是普通业务参数，不会在不同功能载荷中始终固定不变；而单行 Webshell 的密码参数恰恰具备这种特征。

### 3. 为什么不是 `/lang.php` 的密文流量

`/lang.php` 的确也存在大量 200 响应请求，但其请求体为整体加密后的密文块，例如：

```text
cgeX+iSvTbVeKnFcHXywf4E7HUlDiXNN...
```

此类流量只能证明存在加密 Webshell 管理行为，却无法仅凭访问日志直接还原密码。题面问的是“根据日志数据分析并找出攻击者真实攻击载荷密码”，因此应优先选择**在日志中可以直接、稳定、重复验证的真实口令字段**，即 `/font.php` 的固定参数名。

## 四、结论

综合成功状态码、稳定复用特征以及载荷结构，030 题最终答案应为：

```text
flag{5wxBnp5iX879uyP1d2js3MIOVrYFS76U}
```

## 五、补充说明

本题最容易误判的地方，在于把所有加密流量都当成最终答案线索。实际上，真正具有“密码”语义且可被日志直接验证的，是单行 Webshell 的固定 POST 参数名。相比之下，`/lang.php` 仅能证明攻击者后续进行了加密会话控制，但日志本身不足以直接恢复其内部口令，因此不应作为本题最终提交值。
