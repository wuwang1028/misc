# 数据库安全加固 WP

## 一、题目概述

本题的正式题名为 **数据库安全加固**。从题面线索与前期记录可知，这是一道典型的数据库安全整改题，要求选手在给定的在线终端环境中完成三项具体处置：**修改 root 密码、删除 test 数据库、禁止 root 用户远程登录**。题目同时特别提示不能删除 MySQL 中的 `check` 用户，否则会导致判题异常。

这意味着本题并不是让选手继续利用数据库漏洞获取敏感信息，而是要求在尽量不破坏现有业务和裁判逻辑的前提下，对一个存在弱口令、默认库残留和远程高权限暴露问题的 MariaDB 实例完成最小化加固。换言之，题目考察的是 **数据库基线整改能力**，而不是单纯的漏洞利用能力。

## 二、环境识别与基础画像

初始登录身份为浏览器在线终端中的普通用户 `ctf`。实例内运行的是 MariaDB 10.6.14，监听端口为 `3306`。`/home/ctf/README.md` 中只有两条非常关键的约束：其一，服务异常时可以销毁并重建容器；其二，**不要删除 MySQL 中的 `check` 用户**。

| 项目 | 结果 |
| --- | --- |
| 题目名称 | `数据库安全加固` |
| 终端登录身份 | `ctf/ctf` |
| 数据库类型 | MariaDB 10.6.14 |
| 监听端口 | `3306` |
| 关键约束 | 不可删除 `check` 用户 |
| 目标动作 | 改 root 密码、删 test 库、禁 root 远程登录 |
| 判题方式 | 满足条件后生成 `/flag` |

从配置文件与数据库探测结果来看，当前实例存在三个非常明确的安全问题。第一，MariaDB 对外绑定地址为 `0.0.0.0`，允许远程连接。第二，`root@'%'` 账户存在，而且与本地 `root@localhost` 使用同一个口令哈希。第三，默认 `test` 数据库仍然保留。这三点与题面要求完全对应，说明题目本身已经直接给出了加固目标，关键在于如何准确落实且不误伤 `check` 用户。

## 三、初始排查与关键突破

最初直接尝试 `mysql -uroot` 连接时，会收到 `ERROR 1698 (28000): Access denied for user 'root'@'localhost'`，说明 root 并不是空口令登录。然而，继续对常见口令进行有限枚举后，很快确认 **root 的当前密码为 `root`**。无论通过 TCP 连接 `127.0.0.1:3306`，还是通过本地 socket 方式登录，只要使用 `-uroot -proot` 都可以成功进入数据库。

这一点是整个题目的关键突破。因为它不仅证明 root 存在弱口令问题，也使后续所有整改动作都可以直接在数据库内部完成，而不需要依赖系统级提权或服务级配置修改。

与此同时，继续测试其他账户时还发现一个有价值的现象：`admin` 用户可以**无密码本地登录**，但权限极其有限，只能看到 `information_schema` 和 `test` 两个库，`current_user()` 结果显示为 `@localhost`。这表明环境中存在一个低权限、近似匿名能力的本地账号，不过它并不是本题判题重点。真正影响判题的仍然是 root 弱口令、远程 root 暴露和 test 库残留三项问题。

| 检查项 | 初始结果 | 结论 |
| --- | --- | --- |
| `mysql -uroot` | 拒绝访问 | root 不是空口令 |
| `mysql -uroot -proot` | 成功登录 | root 弱口令为 `root` |
| `show databases;` | 包含 `test` | 默认测试库未删除 |
| `select user,host from mysql.user;` | 存在 `root@'%'` | root 允许远程登录 |
| `bind-address` | `0.0.0.0` | 数据库对外监听 |
| `check` 用户 | `check@%` 与 `check@localhost` 均存在 | 判题依赖账户，不可删除 |

## 四、证据链分析

使用 root 成功登录后，读取 `mysql.user` 表和相关授权信息，可以看到如下几条核心事实：本地 root 与远程 root 都启用了 `mysql_native_password`；本地 root 的认证哈希与 `root@'%'` 相同；`check` 用户同时存在于 `%` 和 `localhost`；系统仍然保留 `test` 数据库；数据库绑定地址为 `0.0.0.0`。

这些事实可以拼成一条完整的风险链。

| 证据编号 | 证据内容 | 证明作用 |
| --- | --- | --- |
| 1 | `mysql -uroot -proot` 可成功登录 | 证明 root 存在弱口令 |
| 2 | `show databases;` 结果包含 `test` | 证明默认测试数据库未删除 |
| 3 | `mysql.user` 中存在 `root@'%'` | 证明 root 可远程登录 |
| 4 | `bind-address = 0.0.0.0` | 证明数据库对所有网卡开放监听 |
| 5 | `mysql.user` 中保留 `check@%` 与 `check@localhost` | 说明判题依赖 `check`，不可误删 |
| 6 | 修改后重新查询 `mysql.user` 与数据库列表 | 可直接验证加固是否生效 |
| 7 | `/flag` 生成 | 证明裁判认可整改结果 |

从实际整改逻辑来看，题目真正要求的“禁止 root 用户远程登录”并不一定要求关闭 MySQL 的所有远程访问能力，而是要求**至少移除远程 root 入口**。也就是说，只删除 `root@'%'`，同时保留 `check` 用户和本地 root 管理入口，就可以在不破坏判题逻辑的前提下满足要求。

## 五、加固思路与实际处置

在确认 root 口令为 `root` 之后，最稳妥的处置方式就是直接使用数据库管理员权限一次性完成三项整改：第一，修改 `root@localhost` 的密码；第二，删除 `test` 数据库；第三，删除 `root@'%'` 账户并刷新权限。考虑到题目只要求完成安全整改，而不是恢复复杂业务，因此这种数据库内部最小变更路径既直接，也最符合判题预期。

本次实际执行的整改语句如下：

```sql
ALTER USER 'root'@'localhost' IDENTIFIED BY 'Xctf2026Db_9';
DROP DATABASE test;
DROP USER 'root'@'%';
FLUSH PRIVILEGES;
```

这里特意没有去删除 `check` 用户，也没有去关闭整个数据库监听地址，因为题目明确提醒 `check` 用户不能删除，而删除 `root@'%'` 已经能够满足“禁止 root 用户远程登录”的要求。与此同时，新的 root 密码选择了一个明显强于原弱口令的复杂字符串，用于满足“修改 root 密码”的要求。

## 六、修复后验证

执行整改语句后，立即使用新密码 `Xctf2026Db_9` 重新登录数据库进行验证。验证结果表明：`root@localhost` 仍可正常管理数据库，`test` 库已经不存在，`root@'%'` 账户已被删除，而 `check` 用户仍然保留。此时数据库中仅剩与业务或判题有关的必要账户与系统库。

| 验证项 | 修复后结果 | 说明 |
| --- | --- | --- |
| root 新密码登录 | 成功 | root 密码修改已生效 |
| `show databases;` | 不再包含 `test` | 默认测试库已删除 |
| `select user,host from mysql.user;` | 不再出现 `root@'%'` | 已禁止 root 远程登录 |
| `check` 用户 | 仍存在 | 未破坏裁判依赖 |
| `/flag` | 已生成 | 题目判定成功 |

需要说明的是，在第一次完成整改后，实例连接曾出现短暂波动，再次确认时发现此前修改结果仍然保留，且 `/flag` 已成功生成。这说明加固动作本身已经被裁判正常识别，连接波动只是靶场实例在线终端的临时状态问题，并不影响题目最终通过。

## 八、题目本质总结

这道题的本质，是把真实运维基线中的三个常见风险组合在一起：**弱口令管理账户、默认测试库残留、远程高权限账号暴露**。题目没有设置复杂的利用链，也没有要求深入研究数据库内核机制，而是要求选手准确识别出真正应该整改的对象，并且避免在整改过程中误删判题依赖账户。

从训练价值来看，本题非常适合作为数据库安全基线入门题。它要求选手不仅会“连进去”，还要理解什么修改属于真正的加固，什么修改又会破坏业务或裁判。例如，若一味追求“更安全”而删除 `check` 用户，反而会导致题目无法通过；而只要围绕题目要求完成 **改 root 密码、删 test 库、删 root 远程入口** 三件事，就能准确命中判题条件。

更进一步说，这道题也体现了应急和整改工作的一个核心原则：**最小必要变更优于盲目大范围改造**。在不知道裁判全部内部逻辑的情况下，优先做题面明确要求的最小整改，并保留题面特别提醒不能动的对象，往往才是最稳妥的解法。
