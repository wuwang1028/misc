# 系统后门排查复现方案

## 一、复现目标

本题的复现目标不是单纯复刻一个带后门的 Linux 容器，而是要还原出**“任意用户登录即触发后门连接”**这一系统级持久化场景，并配套一套能在清理完成后自动放出 `/flag` 的判题机制。

因此，后续真正落地该题时，至少需要完整复现以下四个能力点：第一，存在一个可通过 ttyd 登录的 Linux 主机；第二，系统全局登录配置文件被植入恶意触发项；第三，Web 目录下存在实际被调用的隐藏后门脚本；第四，裁判脚本能够在参赛者完成清理后自动生成 `/flag`。

## 二、建议环境架构

从本题真实解法来看，采用**单容器结构**即可满足训练需求，没有必要设计多机拓扑。推荐环境组成如下。

| 组件 | 作用 | 建议实现 |
| --- | --- | --- |
| Alpine 或 Ubuntu 基础系统 | 提供 Linux 主机环境 | Docker 容器 |
| ttyd | 提供 Web 终端入口 | 前台启动 `login` 或 shell |
| `ctf` 用户 | 选手登录账号 | 用户名密码均为 `ctf` |
| `/etc/profile` | 全局登录触发点 | 预埋恶意调用逻辑 |
| `.Backdoor.php` | 实际恶意载荷 | 放置于 Web 目录隐藏路径 |
| 判题脚本 `judge.py` 或 `judge.sh` | 轮询检查修复状态 | 满足条件后写出 `/flag` |

## 三、初始环境设计

为了让题目具备明确的排查路径，建议把初始状态设置为“攻击者已经完成持久化，但还没有把环境做得过于复杂”。

### 3.1 系统账户设计

建议创建普通用户 `ctf`，密码也设置为 `ctf`，用于 Web 终端登录。这样既便于训练，也与原题体验一致。

### 3.2 关键文件布局

建议按以下方式布置关键文件。

| 路径 | 初始状态 | 设计目的 |
| --- | --- | --- |
| `/etc/profile` | 含恶意调用语句 | 对应“任意用户登录即触发” |
| `/var/www/html/application/model/.Backdoor.php` | 存在且可执行 | 作为被调用的隐藏后门文件 |
| `/root/check/judge.py` | 周期运行 | 检查环境是否已清理 |
| `/flag` | 初始不存在 | 由裁判脚本动态生成 |
| `/readflag` | 初始不可直接执行 | 作为干扰项，增加真实感 |

### 3.3 `/etc/profile` 中的恶意逻辑建议

可以在 `/etc/profile` 末尾增加一条明确的恶意调用，例如：

```bash
/var/www/html/application/model/.Backdoor.php >/dev/null 2>&1 &
```

如果希望更贴近原题的运行态，还可以在容器启动脚本中额外拉起一个持续循环：

```bash
su ctf -c 'cd /tmp && while true; do /var/www/html/application/model/.Backdoor.php; sleep 10; done;' &
```

这样做的好处是：参赛者即使已经看到了恶意文件，也仍需意识到**运行中的后门进程同样属于处置对象**。

## 四、恶意样本设计建议

`.Backdoor.php` 不需要实现真正的复杂木马功能，但应当具备足够的“恶意脚本特征”，让参赛者在排查时能明确判断其异常性。推荐至少满足以下几点。

| 要素 | 建议 |
| --- | --- |
| 文件名 | 以点开头，如 `.Backdoor.php` |
| 所在位置 | 混入正常 Web 项目路径，例如 `application/model/` |
| 行为 | 写日志、发起连接、输出异常内容三选一即可 |
| 权限 | 允许 `ctf` 用户重命名或删除 |

一个最简化的示例可以是：

```php
#!/usr/bin/env php
<?php
file_put_contents('/tmp/backdoor.log', date('c')." trigger\n", FILE_APPEND);
?>
```

同时赋予可执行权限：

```bash
chmod +x /var/www/html/application/model/.Backdoor.php
```

## 五、判题逻辑设计

本题的判题逻辑建议采用**状态判定型**，而不是命令回显型。也就是说，裁判不关注参赛者具体执行了什么命令，只检查环境是否已恢复到目标状态。

推荐的检查条件如下。

| 检查项 | 判定标准 |
| --- | --- |
| `/etc/profile` | 不再包含 `.Backdoor.php` 相关调用 |
| 恶意文件 | 原始 `.Backdoor.php` 不存在，或已被改名隔离 |
| 残留进程 | 不再存在持续执行 `.Backdoor.php` 的循环进程 |
| flag 文件 | 上述条件满足后生成 `/flag` |

判题脚本可以按下述逻辑实现：

```bash
#!/bin/sh
while true; do
  bad_profile=0
  bad_file=0
  bad_proc=0

  grep -q 'Backdoor.php' /etc/profile 2>/dev/null && bad_profile=1
  [ -f /var/www/html/application/model/.Backdoor.php ] && bad_file=1
  ps aux 2>/dev/null | grep -v grep | grep -q '/var/www/html/application/model/.Backdoor.php' && bad_proc=1

  if [ "$bad_profile" -eq 0 ] && [ "$bad_file" -eq 0 ] && [ "$bad_proc" -eq 0 ]; then
    echo 'xctf{REPLACE_WITH_REAL_FLAG}' > /flag
    chmod 777 /flag
    break
  fi

  sleep 10
done
```

这种判题方式与真实题目行为非常接近。它会迫使参赛者理解“配置、文件、进程”三者缺一不可，而不是只删除某一个表面异常点。

## 六、启动逻辑设计

建议使用统一启动脚本 `start.sh` 负责拉起各组件，包含以下任务：

| 启动项 | 作用 |
| --- | --- |
| 初始化 `ctf` 用户与密码 | 提供登录能力 |
| 启动 ttyd | 暴露 Web 终端入口 |
| 启动恶意循环任务 | 模拟持续性后门 |
| 启动判题脚本 | 轮询清理结果 |

示意性启动脚本如下：

```bash
#!/bin/sh
set -e

if ! id ctf >/dev/null 2>&1; then
  adduser -D ctf
  echo 'ctf:ctf' | chpasswd
fi

chmod 777 /etc/profile
chmod +x /var/www/html/application/model/.Backdoor.php
chmod 700 /readflag

su ctf -c 'cd /tmp && while true; do /var/www/html/application/model/.Backdoor.php; sleep 10; done;' &
/root/check/judge.sh &
/usr/sbin/ttyd -p 18080 login
```

如果后续采用 Ubuntu 镜像，也可以把用户初始化和 ttyd 启动逻辑替换为对应发行版命令，但整体结构保持不变即可。

## 七、参赛者预期解法

从训练设计角度看，参赛者的标准解法应包含以下几个连续动作：先通过 ttyd 登录；随后根据题面“任意用户登录触发”联想到 `/etc/profile`；接着确认 `.Backdoor.php` 为实际恶意载荷；然后移除 `/etc/profile` 中的恶意调用、隔离 `.Backdoor.php` 文件，并结束遗留循环进程；最后等待裁判脚本生成 `/flag`。

这一路径非常适合作为**Linux 持久化后门排查入门题**，因为它考察的是系统知识、排障习惯和最小修复能力，而不是偏漏洞利用的技巧。

## 八、建议的目录组织

若后续要把该题真正补齐为可运行环境，建议在仓库中采用如下结构：

| 路径 | 内容 |
| --- | --- |
| `系统后门排查/Dockerfile` | 基础镜像、依赖安装、用户创建 |
| `系统后门排查/start.sh` | 启动 ttyd、恶意循环与判题脚本 |
| `系统后门排查/judge.sh` | 状态轮询与 `/flag` 生成逻辑 |
| `系统后门排查/init/profile` | 含恶意调用的初始 `/etc/profile` 模板 |
| `系统后门排查/init/.Backdoor.php` | 隐藏后门样本 |
| `系统后门排查/README.md` | 题目说明、账号与触发规则 |

## 九、落地复现时的注意事项

本题在真正制作时有三个地方需要特别注意。第一，`/etc/profile` 必须确保在登录时一定会被加载，否则题面描述与真实行为会不一致；第二，恶意文件对 `ctf` 用户必须可处置，否则题目会从“应急响应题”意外变成“提权题”；第三，判题脚本应避免只检查文件是否存在，而忽略残留进程，否则题目训练价值会明显下降。

换言之，这道题最值得保留的，不是某个单点文件本身，而是**登录触发、恶意载荷、运行态残留、裁判异步放旗**这四者共同构成的场景闭环。
