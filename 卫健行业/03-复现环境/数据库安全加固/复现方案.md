# 数据库安全加固复现方案

## 一、复现目标

本题的复现重点，不是单纯搭建一个能登录的 MariaDB 服务，而是要完整还原一个**同时存在 root 弱口令、默认 test 数据库残留、root 远程登录开放**的数据库整改场景，并保留一个供裁判使用的 `check` 账户。只有把这几项因素同时保留下来，选手才能按照原题思路完成最小必要变更，并在不破坏判题链路的前提下触发放旗。

换言之，这道题的训练价值并不在于数据库攻击，而在于数据库基线修复：选手需要识别哪些对象必须删、哪些对象必须留、哪些能力应该收紧但不能一刀切地全部关闭。

## 二、建议环境架构

从解题过程来看，本题采用**单容器数据库整改场景**即可复现，不需要复杂的多主机联动。核心组件建议如下。

| 组件 | 作用 | 建议实现 |
| --- | --- | --- |
| Linux 基础镜像 | 提供运行环境 | Ubuntu 或 CentOS 容器 |
| MariaDB | 作为待整改数据库 | 10.5 或 10.6 版本 |
| `ctf` 用户 | 提供在线终端登录身份 | 用户名密码均为 `ctf` |
| ttyd | 提供 Web 终端入口 | 与原题体验一致 |
| 判题脚本 | 定期检查数据库整改状态 | 成功后生成 `/flag` |

这种设计的优点在于非常直接。选手登录后就能围绕数据库账户、数据库列表和授权对象展开排查，不会被无关 Web 应用或系统提权链分散注意力。

## 三、初始环境设计

### 3.1 用户与访问入口

建议创建普通用户 `ctf`，密码同样设置为 `ctf`，并通过 ttyd 暴露在线终端。登录后默认目录可以放在 `/home/ctf`，同时提供一份 README，用于提示选手约束条件和题目边界。

建议在 `/home/ctf/README.md` 中至少写入以下两点：第一，服务异常时可重建容器；第二，**不要删除 MySQL 中的 `check` 用户**。这条提示非常重要，因为它直接决定了题目是否具备“最小变更整改”的训练属性。

### 3.2 MariaDB 初始状态

MariaDB 应被初始化为一个“看似可用、但基线明显不合规”的状态。推荐的关键初始条件如下。

| 项目 | 初始状态 |
| --- | --- |
| `root@localhost` | 使用弱口令 `root` |
| `root@'%'` | 存在，且同样使用弱口令 |
| `check@localhost` | 存在，用于本地判题访问 |
| `check@'%'` | 存在，用于远程或 TCP 判题访问 |
| `test` 数据库 | 保留 |
| `bind-address` | `0.0.0.0` |

其中，是否一定要把 `bind-address` 设为 `0.0.0.0` 可以根据环境决定，但建议保留，因为这能更直观地体现“远程 root 暴露”的风险背景。不过判题本身不必强制要求选手关闭全局监听，只要 root 不再允许远程登录即可。

### 3.3 示例初始化 SQL

为了与原题行为一致，可以在容器启动时执行如下初始化语句：

```sql
CREATE DATABASE IF NOT EXISTS test;

CREATE USER IF NOT EXISTS 'root'@'%' IDENTIFIED BY 'root';
ALTER USER 'root'@'localhost' IDENTIFIED BY 'root';
GRANT ALL PRIVILEGES ON *.* TO 'root'@'%' WITH GRANT OPTION;
GRANT ALL PRIVILEGES ON *.* TO 'root'@'localhost' WITH GRANT OPTION;

CREATE USER IF NOT EXISTS 'check'@'localhost' IDENTIFIED BY 'Check@123';
CREATE USER IF NOT EXISTS 'check'@'%' IDENTIFIED BY 'Check@123';
GRANT SELECT ON *.* TO 'check'@'localhost';
GRANT SELECT ON *.* TO 'check'@'%';

FLUSH PRIVILEGES;
```

这里的关键不是 `check` 用户必须有什么高权限，而是它必须存在，并被裁判逻辑依赖。只有这样，选手才需要在整改时主动避开它。

## 四、题目故障与风险点设计

本题的初始风险点应当同时满足以下三个条件。第一，管理员 root 使用显而易见的弱口令。第二，默认 `test` 数据库仍然残留。第三，远程 root 入口开放。这样一来，题目要求与实际环境能够一一对应，选手的排查思路也会非常清晰。

| 风险项 | 初始表现 | 对应整改动作 |
| --- | --- | --- |
| root 弱口令 | `mysql -uroot -proot` 可登录 | 修改 root 密码 |
| 默认测试库残留 | `show databases;` 含 `test` | 删除 `test` 数据库 |
| root 远程登录 | `mysql.user` 中有 `root@'%'` | 删除或禁用 `root@'%'` |
| 误删关键账户风险 | `check` 用户参与判题 | 必须保留 `check` |

需要注意的是，题目应当设计为“**禁止 root 远程登录**”，而不是“完全关闭数据库远程访问”。原因在于后者容易引导选手做过度整改，例如直接改监听地址、停数据库或误删多个账户，反而弱化了题面要求的精准性。

## 五、判题逻辑设计

本题最适合采用**数据库状态判定型**裁判。裁判不需要关心选手具体执行了什么命令，只需要周期性检查以下四个条件是否同时满足：其一，`root@localhost` 已经不是弱口令 `root`；其二，`test` 数据库已删除；其三，`root@'%'` 不存在；其四，`check` 用户仍然保留。

推荐的判题条件可以总结如下。

| 检查项 | 判定标准 |
| --- | --- |
| root 密码 | 不再等于初始弱口令 `root` |
| `test` 数据库 | 不存在 |
| `root@'%'` | 不存在 |
| `check` 用户 | 仍存在 |
| `/flag` | 满足条件后由裁判自动生成 |

示意性判题脚本可以写成：

```bash
#!/bin/bash
while true; do
  root_ok=0
  db_ok=0
  remote_ok=0
  check_ok=0

  mysql -uroot -proot -e 'select 1' >/dev/null 2>&1 || root_ok=1
  mysql -uroot -p'NEW_PASSWORD' -Nse "show databases like 'test';" 2>/dev/null | grep -q . || db_ok=1
  mysql -uroot -p'NEW_PASSWORD' -Nse "select 1 from mysql.user where user='root' and host='%';" 2>/dev/null | grep -q . || remote_ok=1
  mysql -uroot -p'NEW_PASSWORD' -Nse "select 1 from mysql.user where user='check';" 2>/dev/null | grep -q . && check_ok=1

  if [ "$root_ok" -eq 1 ] && [ "$db_ok" -eq 1 ] && [ "$remote_ok" -eq 1 ] && [ "$check_ok" -eq 1 ]; then
    echo 'xctf{REPLACE_WITH_REAL_FLAG}' > /flag
    chmod 644 /flag
    break
  fi
  sleep 5
done
```

在正式出题时，不建议把新密码写死在判题脚本里。更稳妥的做法是读取 `root@localhost` 的哈希，判断其是否与初始 `root` 的哈希不同，同时确认数据库和用户对象状态即可。

## 六、标准解法设计

本题的标准解法应该非常明确：使用 root 弱口令登录数据库，执行最小整改语句，随后等待裁判放旗。推荐作为官方预期的解法如下。

```sql
ALTER USER 'root'@'localhost' IDENTIFIED BY 'Xctf2026Db_9';
DROP DATABASE test;
DROP USER 'root'@'%';
FLUSH PRIVILEGES;
```

这一路径的优点在于完全命中题意，同时不会影响 `check` 用户。若选手采用等效方式，例如 `RENAME USER`、`SET PASSWORD` 或 `DELETE FROM mysql.user` 后再 `FLUSH PRIVILEGES`，原则上也可视为通过，只要最终数据库状态满足判题条件即可。

## 七、启动逻辑设计

建议使用统一的 `start.sh` 完成数据库初始化、README 写入、判题脚本拉起和 ttyd 暴露。推荐职责如下。

| 启动项 | 作用 |
| --- | --- |
| 启动 MariaDB | 提供待整改数据库 |
| 执行初始化 SQL | 设置 root 弱口令、`test` 库与 `check` 用户 |
| 写入 `/home/ctf/README.md` | 提示不可删除 `check` |
| 启动判题脚本 | 周期检测整改状态 |
| 启动 ttyd | 提供在线终端 |

一个示意性的 `start.sh` 可以写成：

```bash
#!/bin/bash
set -e

service mariadb start
mysql <<'SQL'
CREATE DATABASE IF NOT EXISTS test;
ALTER USER 'root'@'localhost' IDENTIFIED BY 'root';
CREATE USER IF NOT EXISTS 'root'@'%' IDENTIFIED BY 'root';
GRANT ALL PRIVILEGES ON *.* TO 'root'@'%' WITH GRANT OPTION;
CREATE USER IF NOT EXISTS 'check'@'localhost' IDENTIFIED BY 'Check@123';
CREATE USER IF NOT EXISTS 'check'@'%' IDENTIFIED BY 'Check@123';
GRANT SELECT ON *.* TO 'check'@'localhost';
GRANT SELECT ON *.* TO 'check'@'%';
FLUSH PRIVILEGES;
SQL

cat >/home/ctf/README.md <<'EOF'
1. 服务异常时，可销毁当前容器，再次生成容器
2. 请勿删除mysql中的check用户，否则会导致check异常
EOF

/root/check/judge.sh &
/usr/bin/ttyd -p 18080 login
```

| `数据库安全加固/Dockerfile` | MariaDB、ttyd 与基础依赖安装 |

## 九、制作时的注意事项

本题制作时有三点尤其要注意。第一，必须确保 root 弱口令真实可用，否则选手无法进入数据库，自然也无法开展整改。第二，必须让 `check` 用户真实参与判题，而不是只写在题面里，否则“不要删除 check”就会失去意义。第三，判题逻辑应聚焦最终数据库状态，而不是强依赖某条固定命令，这样题目会更稳健，也更贴近真实运维整改场景。

归根结底，这道题最值得保留的不是某个具体密码，而是 **弱口令 root、默认测试库、远程 root 暴露、关键判题账户保留** 共同组成的整改闭环。只要这个闭环被正确复刻，这道题就能稳定训练选手的数据库安全基线修复能力。
