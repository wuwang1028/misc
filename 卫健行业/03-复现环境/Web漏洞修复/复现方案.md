# Web漏洞修复复现方案

## 一、复现目标

本题的复现目标，不是机械地还原某个现成的 Spring Boot 项目，而是完整保留这道题真正有训练价值的闭环：**一个 Java 文件上传服务存在文件控制不足问题，导致上传文件名可被利用为路径逃逸或任意路径写入；选手需要在不破坏基本业务行为的前提下修复上传控制逻辑；修复成功后由裁判自动生成 `/flag`**。[1] [2]

换言之，复现环境至少需要满足以下几点：终端用户为 `ctf/ctf`；服务以 JAR 形式部署；业务入口仍然是 `GET /` 上传表单与 `POST /doUpload` 上传接口；初始版本对文件名路径控制不足；选手只能通过替换 JAR 并执行 `sudo /restart-web.sh` 完成修复；当上传路径控制满足预期后，裁判脚本自动生成 `/flag`。[1] [3]

| 复现目标 | 说明 |
| --- | --- |
| 保留题目形态 | 必须是“在线终端 + JAR 替换 + 服务重启”模式 |
| 保留业务入口 | 根路径表单与 `/doUpload` 上传接口都应存在 |
| 保留漏洞本质 | 初始版本需要体现文件控制不足导致的路径逃逸风险 |
| 保留判题机制 | 修复成功后自动生成 `/flag` |

## 二、建议环境架构

从本题真实解法来看，采用**单容器 Java 服务场景**即可较好复刻核心训练价值，不需要再额外引入数据库等复杂依赖。建议组件包括 Linux 基础镜像、OpenJDK 运行时、ttyd、`ctf` 普通用户、上传服务 JAR、重启脚本与周期判题脚本。[1] [2]

| 组件 | 作用 | 建议实现 |
| --- | --- | --- |
| Linux 基础镜像 | 提供统一运行环境 | Debian 或 Ubuntu 容器 |
| OpenJDK | 运行上传服务 JAR | JDK 17 或兼容版本 |
| ttyd | 提供在线终端入口 | 维持与原题一致的操作习惯 |
| `ctf` 用户 | 提供普通登录身份 | 用户名密码均为 `ctf` |
| 初始漏洞版 JAR | 作为待修复目标 | 暴露路径控制不足问题 |
| `/restart-web.sh` | 控制替换与重启 | 从 `/home/ctf/` 复制 JAR 到 `/opt/app/` |
| `judge.sh` | 周期验证修复状态 | 满足条件后生成 `/flag` |

这种设计的优点在于，题目的关键难点——识别漏洞本质、在受限权限下交付修复包、通过标准脚本重启并等待判题——都能被完整保留下来，而不会被环境噪声稀释。

## 三、初始环境设计

建议把运行目录固定为 `/opt/app`，把选手可投递的替换目录固定为 `/home/ctf`。这样既符合原题逻辑，也能清晰体现“用户目录中的 JAR 只是候选制品，真正运行的服务以 `/restart-web.sh` 拷贝上线为准”这一特点。[1]

| 路径 | 作用 | 初始状态 |
| --- | --- | --- |
| `/opt/app/java-upload-fix-1.0-SNAPSHOT.jar` | 运行中的漏洞版服务 | 容器启动后默认执行 |
| `/opt/app/upload/` | 服务上传目录 | 初始存在，可写 |
| `/home/ctf/README.md` | 题目说明 | 明确要求下载、修补、回传、重启 |
| `/home/ctf/java-upload-fix-1.0-SNAPSHOT.jar` | 选手回传位置 | 初始不存在 |
| `/restart-web.sh` | 重启脚本 | 负责复制替换包并拉起服务 |
| `/root/check/judge.sh` | 判题脚本 | 检查修复效果并放旗 |
| `/flag` | 最终答案文件 | 初始不存在 |

与此同时，建议让容器启动脚本默认执行漏洞版 JAR，并额外启动 ttyd 与判题脚本。这样选手首次进入环境时就能直接观察到在线终端与服务已运行的状态。

## 四、漏洞版业务逻辑设计

为了使题目具有真实训练意义，初始漏洞版服务应保留一个足够接近正式题面的上传行为：根路径返回一个上传表单，表单字段名为 `file`，提交到 `/doUpload`；服务要求上传文件具备 `.jpg` 等有限后缀，但在保存文件时**没有正确限制文件名中的绝对路径、路径分隔符或目录穿越片段**。这类设计既能解释为什么普通文本上传会失败，也能让漏洞点精准聚焦到“文件控制不足”。[1]

一个典型的脆弱实现方式，是在做了“.jpg 后缀存在”的浅层检查后，仍然直接使用用户提供的 `filename` 与目标目录拼接，或错误地允许绝对路径、`../` 路径片段通过。这样就会出现以下训练效果：选手不能简单上传任意文件，但如果理解了文件路径控制机制，仍然可以把文件写出上传目录之外。

| 检查项 | 初始漏洞版应表现出的现象 |
| --- | --- |
| `GET /` | 返回上传表单 |
| `POST /doUpload` 正常 `.jpg` | 可上传并得到写入路径回显 |
| `POST /doUpload` 含绝对路径文件名 | 初始版本错误地允许或处理不当 |
| `POST /doUpload` 含 `../` 文件名 | 初始版本错误地允许或处理不当 |

## 五、修复版预期行为设计

修复版的要求不应是删除上传功能，而应是在保留原业务入口与上传能力的前提下，把文件控制逻辑收敛到安全边界。就本题实际解法而言，最稳妥的做法包括以下四步：先统一路径分隔符，再只保留最终基名，然后拒绝包含 `..`、`/`、`\` 的文件名，最后在拼接出目标路径后使用 `normalize()` 与 `startsWith(UPLOAD_DIR)` 确认路径没有逃逸上传目录。[2]

| 修复要求 | 预期效果 |
| --- | --- |
| 仅允许合法基名 | 阻断目录拼接与绝对路径写入 |
| 显式拒绝 `..` | 阻断目录穿越 |
| 固定上传目录 | 避免写到任意系统路径 |
| 保留 `.jpg` 上传能力 | 不破坏原业务基本功能 |

如果要在仓库中提供一个“修复版参考实现”，可以直接复用本次实战中整理出的 `SimpleUploadServer.java` 作为最小兼容实现。它已经覆盖了根路径表单、`/doUpload`、`.jpg` 约束、路径规范化与上传目录约束等关键逻辑。[2]

## 六、判题逻辑设计

本题建议采用**本地探测型裁判**，而不是只检查某个文件是否存在。裁判脚本应从 `127.0.0.1:8080` 发起至少两类请求：一类是正常 `.jpg` 上传，确认服务仍可用；另一类是带路径逃逸特征的非法文件名上传，确认其已被拒绝。只有当“业务仍在”与“路径逃逸被阻断”同时满足时，才生成 `/flag`。这比只检查文件内容更能体现本题的训练目标。[1] [2]

示意性的判题逻辑可以设计为：先上传一个正常 `test.jpg`，若接口成功返回上传目录内路径，则记为业务可用；再尝试上传如 `/tmp/evil.jpg` 或 `../../tmp/evil.jpg` 这样的文件名，若接口返回 `nonono`，且对应目标路径未新增，则判定修复成功，随后写入 `/flag`。

```bash
#!/bin/sh
while true; do
  printf 'ok' >/tmp/ok.bin

  good=$(curl -s -m 5 -F "file=@/tmp/ok.bin;filename=test.jpg" http://127.0.0.1:8080/doUpload 2>/dev/null)
  bad=$(curl -s -m 5 -F "file=@/tmp/ok.bin;filename=/tmp/evil.jpg" http://127.0.0.1:8080/doUpload 2>/dev/null)

  if echo "$good" | grep -q '/opt/app/upload/test.jpg' && [ "$bad" = 'nonono' ]; then
    echo 'xctf{REPLACE_WITH_REAL_FLAG}' >/flag
    chmod 644 /flag
    break
  fi

  sleep 5
done
```

## 七、启动与重启逻辑设计

建议使用统一 `start.sh` 拉起漏洞版 JAR、判题脚本与 ttyd，并在容器初始化时创建 `ctf` 用户。与此同时，`/restart-web.sh` 应严格模拟原题逻辑：只有当 `/home/ctf/java-upload-fix-1.0-SNAPSHOT.jar` 存在时，才删除 `/opt/app` 中的旧版本并复制新包，然后以 root 重启服务。[1]

| 脚本 | 职责 |
| --- | --- |
| `start.sh` | 初始化目录、拉起漏洞版服务、启动判题与 ttyd |
| `/restart-web.sh` | 用选手回传的 JAR 覆盖运行包并重启 |
| `judge.sh` | 周期验证修复状态并放旗 |

一个示意性的 `restart-web.sh` 可以保持如下形态。

```bash
#!/bin/bash
set -e
pkill -f 'java -jar /opt/app/java-upload-fix-1.0-SNAPSHOT.jar' || true
if [ -f /home/ctf/java-upload-fix-1.0-SNAPSHOT.jar ]; then
  rm -rf /opt/app/*
  mkdir -p /opt/app/upload
  cp /home/ctf/java-upload-fix-1.0-SNAPSHOT.jar /opt/app/java-upload-fix-1.0-SNAPSHOT.jar
  chmod 755 /opt/app/java-upload-fix-1.0-SNAPSHOT.jar
fi
nohup java -jar /opt/app/java-upload-fix-1.0-SNAPSHOT.jar >/tmp/web.log 2>&1 &
```

## 九、落地复现时的注意事项

本题在制作时尤其要注意三点。第一，不能把题目简化成“普通上传功能关闭”一类静态配置题，否则会失去原题强调的业务兼容性。第二，判题逻辑不应只验证非法上传失败，还应同时验证正常 `.jpg` 上传仍可用，否则会鼓励选手通过粗暴关闭接口来取巧。第三，`/restart-web.sh` 与 `/home/ctf`、`/opt/app` 的双目录结构必须保留，因为这正是本题“替换 JAR 并上线”的核心操作链条。[1] [2]

归根结底，这道题最值得保留的不是某个具体框架，而是 **上传表单、文件名路径控制缺陷、受限终端回传修复包、脚本化重启、异步判题放旗** 共同构成的完整闭环。只要这个闭环被正确复刻，本题就能稳定训练选手对上传类路径控制漏洞的识别、修补与交付能力。
