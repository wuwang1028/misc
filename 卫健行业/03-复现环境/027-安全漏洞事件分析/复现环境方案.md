# 环境类赛题整体解决方案

基于当前已经整理出的题目总表，环境类赛题共 **9 道**，其特点是**没有现成附件**、通常需要在平台启动靶场后进行在线排查、修复、加固或溯源。因此，这一部分工作的核心不只是“做出答案”，而是把每一道题抽象为一类可复用的安全运维/应急响应场景，再为其构建一套**本地可复现的 Docker 环境**、**标准化处置流程**与**最终 WriteUp 模板**。这样做的结果是：后续不仅能拿到题目答案，也能形成一批可重复演练的本地实验环境。

## 一、总体目标

本阶段的目标不是立即逐题提交平台答案，而是先建立一套可执行、可交付、可复现的整体方案。最终交付物将包括三部分：第一部分是每道环境题的**本地 Docker 复现环境**；第二部分是每道题的**处置脚本、检查命令与修复步骤**；第三部分是按统一结构撰写的 **WriteUp**，其中要明确记录题目现象、排查思路、关键证据、修复方式、验证方法与复盘结论。

## 二、环境类题目与建议处置方向

| 序号 | 题目 | 题型定位 | 核心目标 | 本地复现方向 |
|---|---|---|---|---|
| 1 | 027-安全漏洞事件分析 | 漏洞利用溯源 | 识别攻击入口、利用链与落地痕迹 | 构建存在已知漏洞的 Web 服务与访问日志 |
| 2 | 慢SQL优化 | 数据库性能优化 | 定位慢查询、优化索引或 SQL 结构 | MySQL + 慢查询日志 + 样例数据表 |
| 3 | 主机基线加固 | Linux 基线安全 | 修复弱口令、权限、服务暴露等基线问题 | Ubuntu/CentOS 容器 + 故意保留弱配置 |
| 4 | 系统后门排查 | 主机入侵排查 | 识别持久化后门、异常账户、恶意启动项 | 带 cron / systemd / SSH 后门痕迹的 Linux 容器 |
| 5 | 服务后门排查 | 服务级持久化 | 排查定时任务、守护进程、异常端口 | 带恶意定时任务与伪装进程的服务容器 |
| 6 | Web服务故障处理 | Web 运维故障 | 恢复服务可用性并修正配置错误 | Nginx / PHP / 数据库联动故障场景 |

| 8 | 数据库安全加固 | 数据库安全 | 修复口令、权限、远程访问、危险配置 | MySQL 容器 + 高危授权与弱配置 |
| 9 | 系统服务加固 | 中间件加固 | 加固 Redis 等服务的监听、认证与危险命令 | Redis 容器 + 未授权访问与高危配置 |

## 三、统一实施方法

环境类题目虽然题目名称不同，但实际可以统一为一条标准流水线：先**识别故障/风险现象**，再**收集证据**，随后**定位根因**，最后**修复并验证**。在执行层面，我建议对每道题都采用同一工作流，这会显著减少后续 WriteUp 的组织成本。

| 阶段 | 目标 | 典型动作 | 最终产物 |
|---|---|---|---|
| 1. 场景信息采集 | 明确系统状态与异常现象 | 查看题目描述、服务端口、页面报错、进程、日志、配置文件 | 初始状态记录 |
| 2. 证据固定 | 保存关键线索，防止修复后痕迹丢失 | 导出日志、配置、异常进程信息、文件哈希 | 证据清单 |
| 3. 根因定位 | 判断问题属于漏洞、配置、权限、持久化还是性能 | 交叉分析日志、配置、账户、计划任务、数据库状态 | 根因结论 |
| 4. 修复处置 | 消除问题并恢复业务/安全状态 | 修改配置、清除后门、补丁修复、索引优化、权限收敛 | 修复步骤 |
| 5. 验证回归 | 确认问题不再复现且服务正常 | 重新访问服务、压测、复查日志、运行检查脚本 | 验证结果 |
| 6. 环境封装 | 将题目抽象为可本地复现的 Docker 环境 | 编写 Dockerfile、docker-compose、初始化脚本 | 可复现环境 |

## 四、逐题技术方案

### 1. 027-安全漏洞事件分析

这类题通常要求确认**攻击入口、漏洞类型、攻击者行为轨迹**，因此排查重点不是简单“修好服务”，而是还原一次入侵事件链。我的解法会优先关注 Web 访问日志、应用日志、异常文件落地路径、上传目录、WebShell 痕迹和数据库操作记录。如果平台场景中存在易受攻击的组件，我会在本地复现一个相同漏洞版本的应用，并配套一份模拟攻击流量与受害主机日志。

Docker 复现方案上，建议使用 `nginx + php-fpm + vulnerable app` 三段式结构，并增加 `access.log`、`error.log` 与一份故意残留的恶意文件。这样可以把题目抽象为一类通用的“漏洞利用后溯源”实验。

### 2. 慢SQL优化

这一题的目标不是安全排查，而是**数据库性能诊断**。标准做法是先启用或读取慢查询日志，观察 SQL 扫描行数、执行时间、是否命中索引，再结合 `EXPLAIN` 分析执行计划。通常可从建立复合索引、避免全表扫描、减少回表、改写排序/分页 SQL 等几个方向优化。

本地复现建议使用 `mysql:8` 容器，预置一张数据量足够大的业务表，并故意缺失关键索引，再提供一条高耗时 SQL。交付时应包含优化前后执行计划对比、索引创建语句、性能验证方法。

### 3. 主机基线加固

这类题的核心是**把明显不合规的系统设置收敛到安全基线**。重点检查口令策略、SSH 配置、root 远程登录、文件权限、SUID/SGID 异常、危险服务暴露、历史命令与可疑账户。处置逻辑会以“先盘点、再收敛、后验证”为原则，避免为了拿 flag 直接破坏现场。

Docker 复现时，可以准备一个故意开启弱 SSH 策略、权限混乱、存在危险可写目录的 Linux 容器，并附带一份自动检查脚本，用于验证加固前后差异。

### 4. 系统后门排查

该题更偏向应急响应，关键在于定位**持久化机制**。重点对象包括 `/etc/passwd`、`authorized_keys`、`cron`、`systemd`、登录脚本、`rc.local`、异常环境变量、隐藏文件与可疑 ELF/脚本。必要时还要检查最近修改时间、可疑进程父子关系以及网络连接。

本地复现可预置一个通过 `cron` 或 `systemd service` 自启动的恶意脚本，并在日志中留下少量证据，使排查过程既有线索又不过度明显。

### 5. 服务后门排查

与“系统后门排查”相比，这一题更可能聚焦**服务级别的伪装进程和计划任务**。也就是说，后门未必植入系统关键文件，而可能借某个常驻服务名义存活。排查时要重点观察监听端口、异常守护进程、重复拉起机制、计划任务、服务单元文件与进程启动命令行。

Docker 复现建议采用一个看似正常的业务容器，内部附带伪装成日志收集或同步服务的恶意脚本，并通过 `crond` 定时拉起。交付时应包含“发现异常服务名—追踪文件路径—切断持久化—验证端口消失”的完整链路。

### 6. Web服务故障处理

这类题通常不是“被黑”，而是“服务起不来、页面 502/500、数据库连不上、站点配置错乱”等运维故障。标准步骤是先确认 Web 入口是否可访问，再看 Nginx/Apache 状态，接着检查应用运行时、数据库连接、配置文件、文件权限与端口监听情况。

本地复现可构造 `nginx + php-fpm + mysql` 或 `nginx + flask` 的最小三层应用，并故意制造如上游配置错误、数据库凭据错误、站点根目录错误等问题。WriteUp 中要体现“症状—根因—修复—验证”四段式结构。

### 7. Web漏洞修复

这道题的核心不是利用漏洞，而是**识别漏洞点并完成修补**。因此工作重点在源码审计与配置审计。若题目是上传漏洞，就关注 MIME 校验、后缀白名单、解析链；若是 SQL 注入，就改用参数化查询；若是文件包含，就限制路径与输入源；若是命令执行，就进行严格白名单与参数隔离。

本地复现建议保留一份“漏洞版应用”和一份“修复版应用”，并通过 Docker Compose 分别启动，方便在 WriteUp 中展示修复前后差异。

### 8. 数据库安全加固

从已有截图看，这题与 **MySQL 安全配置**强相关。解决方案会围绕四个重点展开：一是账号口令与匿名账户清理；二是最小权限授权；三是禁止危险远程访问与限制监听；四是调整不安全配置项。必要时还会检查审计日志、数据目录权限、测试库与高危 UDF 风险。

Docker 复现建议使用 MySQL 容器，预置弱口令、`root@'%'`、过宽权限与测试库，随后通过初始化脚本模拟高危初始状态，再编写一份加固脚本实现自动修复。

### 9. 系统服务加固

从你前面的描述看，这题与 **Redis 服务加固**高度相关。其标准处置思路包括：绑定监听地址、启用认证、关闭危险命令重命名或禁用、关闭不必要持久化写入风险、限制受保护模式外暴露，以及确认配置文件与数据目录权限。若题目涉及未授权访问，还需要验证外部是否已无法直接连接。

Docker 复现建议直接使用 Redis 容器，初始状态中保留 `bind 0.0.0.0`、无密码、危险命令未禁用等配置。修复后通过脚本验证未授权访问已失效。

## 五、Docker 复现的统一规范

为了便于后续交付和维护，我建议所有环境题都遵循统一目录结构。这样一方面方便你本地启动与复盘，另一方面便于后续继续扩展更多赛题。

| 路径层级 | 内容 |
|---|---|
| `env_challenges/<题目名称>/Dockerfile` | 单容器场景构建文件 |
| `env_challenges/<题目名称>/docker-compose.yml` | 多容器编排定义 |
| `env_challenges/<题目名称>/init/` | 初始化脚本、样例数据、故障配置、后门植入脚本 |
| `env_challenges/<题目名称>/solution/` | 修复脚本、检查命令、自动验证脚本 |
| `env_challenges/<题目名称>/writeup.md` | 单题 WriteUp |

建议所有题目都具备三个状态：**原始故障态**、**修复完成态**、**自动验证态**。前者用于复现题目，后者用于演示解题正确性，第三者用于批量校验环境是否符合预期。

## 六、推荐实施顺序

从成本和成功率角度考虑，不建议一开始就处理最复杂的溯源类题目，而应先从边界清晰、复现容易的题目入手。这样既能快速产出可交付成果，也能沉淀一套模板，反向支持后续更复杂题目的处理。

| 优先级 | 建议先做的题目 | 原因 |
|---|---|---|
| 第一批 | 数据库安全加固、系统服务加固、慢SQL优化 | 场景边界清晰，Docker 复现最稳定 |
| 第二批 | Web服务故障处理、Web漏洞修复、主机基线加固 | 需要多组件联动，但题目结构较标准 |
| 第三批 | 系统后门排查、服务后门排查、027-安全漏洞事件分析 | 证据链更长，适合在模板成熟后处理 |

## 七、我接下来会如何推进

如果你确认按这套方案执行，我将直接进入逐题实施阶段。具体顺序建议如下：先完成 **数据库安全加固** 与 **系统服务加固** 两题，因为这两题的截图信息相对明确，且 Docker 复现价值最高。完成后，我会交付每题的目录、Docker 文件、初始化脚本、修复脚本、验证方式与 WriteUp，再按同样模板扩展到其余 7 题。

换句话说，当前最合适的下一步不是一次性平推 9 题，而是先打通 **2 题模板**，把复现框架、文档模板和验证方法沉淀下来，再批量复制到剩余题目。这会更稳，也更利于后续维护。
