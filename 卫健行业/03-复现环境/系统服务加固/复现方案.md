# 系统服务加固复现方案

## 一、复现目标

本复现方案的目标，是在可控实验环境中重建《系统服务加固》一题的核心攻击面、判题逻辑与最终修复链条，使后续读者能够独立完成以下三件事：第一，识别 FTP 与 Redis 的组合风险；第二，验证为什么“只修 FTP 仍然无法通过判题”；第三，复现最终有效的修复动作，即 **Redis 启用认证 + FTP 禁止匿名访问**。

## 二、最小环境组成

建议使用 Docker 或轻量虚拟机搭建一个与题目行为尽量接近的实验环境，至少包含如下组件：

| 组件 | 作用 | 最低要求 |
| --- | --- | --- |
| Alpine / Debian 容器 | 提供基础系统环境 | 能运行 `ttyd`、`redis-server`、`vsftpd` |
| `ttyd` | 提供浏览器终端入口 | 监听 `8080`，登录用户为 `ctf` |
| `redis-server` | 模拟内网缓存服务 | 初始状态不启用 `requirepass` |
| `vsftpd` | 模拟 FTP 服务 | 初始状态允许匿名登录，且允许 `PORT` 主动模式 |
| 判题脚本 | 模拟题目后台检查逻辑 | 每 15 秒检查一次 Redis 与 FTP 状态 |

建议建立 `ctf` 用户，并赋予其以下最小操作条件：

1. 可写 `/etc/vsftpd/vsftpd.conf`；
2. 可通过 `sudo /restart-ftp.sh` 重启 FTP；
3. 无法直接获得完整 root shell。

## 三、脆弱态配置建议

为尽量贴近题目原始状态，建议在初始环境中使用以下配置特征：

| 服务 | 初始脆弱态 |
| --- | --- |
| Redis | 不配置 `requirepass` |
| FTP | `anonymous_enable=YES` 或保持匿名可登录效果 |
| FTP | `port_enable=YES` 或保持主动模式可用效果 |
| FTP 配置文件 | 权限放宽，使 `ctf` 可直接编辑 |
| 重启脚本 | `pkill vsftpd` 后重新执行 `vsftpd /etc/vsftpd/vsftpd.conf &` |

如果希望更贴近真实题面，还可以让重启脚本使用**未写绝对路径**的 `vsftpd`，以便复现实战中通过 PATH 解析观察 root 行为的过程。

## 四、判题脚本复刻思路

题目真实判题逻辑可以抽象为如下流程：

1. 连接 `127.0.0.1:6379`；
2. 发送错误 `AUTH aaaaaaaaaaaaaaaaaaaaaaaaaaaaa`；
3. 如果返回“未配置密码”，则判定失败；
4. 如果返回 `WRONGPASS`，继续检查匿名 FTP 登录；
5. 若匿名 FTP 登录失败，则生成 `/flag`。

一个最小化伪代码如下：

```python
import time
import socket
from ftplib import FTP

def check_anonymous_ftp(host):
    try:
        ftp = FTP()
        ftp.connect(host)
        ftp.login()
        ftp.quit()
        return True
    except Exception:
        return False

def check():
    s = socket.socket()
    s.connect(("127.0.0.1", 6379))
    s.send(b"auth aaaaaaaaaaaaaaaaaaaaaaaaaaaaa \n")
    result = s.recv(1024)
    s.close()
    if b"called without any password configured" in result:
        return 0
    elif b"WRONGPASS" in result:
        return 0 if check_anonymous_ftp("127.0.0.1") else 1
    return 0
```

## 五、复现实验步骤

### 1. 验证初始风险

首先确认 Redis 未启用认证，错误 `AUTH` 返回“未配置密码”；然后确认 FTP 匿名登录成功。此时即使只处理一侧风险，也不应直接视为题目通过。

### 2. 仅修 FTP，观察失败

将 FTP 调整为：

- `anonymous_enable=NO`
- `port_enable=NO`

重启 vsftpd 后再次运行判题脚本。此时由于 Redis 仍未启用认证，判题应继续失败。这一步是复现本题最重要的“误导点”。

### 3. 补齐 Redis 认证

在 Redis 中设置：

- `CONFIG SET requirepass Manus123`

之后再测试：

- 错误 `AUTH aaaaaaaaaaaaaaaaaaaaaaaaaaaaa` 应返回 `WRONGPASS`；
- 正确 `AUTH Manus123` 应返回 `+OK`。

### 4. 联合验证

在 Redis 认证和 FTP 匿名禁用同时满足后，运行判题脚本，确认 `/flag` 被生成。

## 七、复盘建议

本题最值得保留的教学价值，不在于 Redis 或 FTP 的单独配置项，而在于**组合判题逻辑带来的误导性**。如果复现实验只保留“禁用匿名 FTP 即可过题”的路径，就会丢失这道题真正的训练重点。建议在教学或仓库呈现中明确强调：

> 本题的真正通过条件，是 **Redis 开启认证** 与 **FTP 禁止匿名登录** 同时成立，而不是只完成表层的 FTP 加固。
