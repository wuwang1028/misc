# Web服务故障处理复现方案

## 一、复现目标

本题的复现目标不是简单搭建一个报错的 PHP 站点，而是要完整还原 **“数据库密码已经迁移到环境变量，但 Web 运行时无法读取该变量，从而导致业务连库失败”** 的真实故障场景。只有同时复刻应用源码、环境变量、PHP-FPM 行为差异、判题接口和放旗逻辑，这道题的训练价值才会完整保留下来。

换言之，复现环境至少需要满足以下几点：题目页面可以正常访问；业务代码表面上已经使用环境变量读取数据库密码；系统层面确实存在 `DATABASE_PASSWORD`；但是通过 Web 请求访问应用时，PHP-FPM 处理进程读不到这个环境变量，导致数据库查询失败；当修复到裁判预期状态后，再自动生成 `/flag`。

## 二、建议环境架构

从真实解法来看，本题采用 **单容器 Web 场景** 即可复现，不需要多主机联动。核心组件建议如下。

| 组件 | 作用 | 建议实现 |
| --- | --- | --- |
| Linux 基础镜像 | 提供运行环境 | Ubuntu 或 Debian 容器 |
| Nginx | 提供 HTTP 服务入口 | 反向代理到 PHP-FPM |
| PHP-FPM | 执行 ThinkPHP 应用 | 保持默认 `clear_env` 行为 |
| MySQL / MariaDB | 提供业务数据 | 预置 `user` 表数据 |
| ThinkPHP 5 应用 | 作为故障业务站点 | 控制器提供 `getuser` 与 `testmysql` |
| `ctf` 用户 | 提供在线终端登录身份 | 用户名密码均为 `ctf` |
| ttyd | 提供 Web 终端入口 | 与题面终端体验保持一致 |
| 判题脚本 | 定期访问校验接口并放旗 | 成功后生成 `/flag` |

这种设计的优点在于，排查路径与原题一致：选手需要同时观察源码、系统环境和运行时行为，而不是把题目简化成一个静态配置题。

## 三、初始环境设计

### 3.1 用户与访问入口

建议创建普通用户 `ctf`，密码同样设置为 `ctf`，并通过 ttyd 暴露 Web 终端。与此同时，Nginx 对外提供业务入口，选手可以直接访问首页和相关接口路径，观察故障表现。

### 3.2 应用目录与关键文件

推荐的关键文件布局如下。

| 路径 | 作用 | 初始状态 |
| --- | --- | --- |
| `/var/www/html/application/database.php` | ThinkPHP 数据库配置 | 使用 `getenv('DATABASE_PASSWORD')` 取密 |
| `/var/www/html/application/index/controller/Index.php` | 业务控制器 | 提供 `getuser()` 查询接口 |
| `/var/www/html/application/index/controller/Check.php` | 判题控制器 | 提供 `testmysql()` 校验接口 |
| `/usr/local/etc/php-fpm.d/www.conf` | PHP-FPM 池配置 | 保持 `;clear_env = no` 为注释状态 |
| `/flag` | 最终答案文件 | 初始不存在 |
| `/readflag` | 干扰项二进制或脚本 | 可保留，也可省略 |
| `/root/check/judge.sh` | 判题脚本 | 周期访问 `testmysql()` |

### 3.3 数据库初始化

建议预置一个简单业务库，例如 `users`，其中包含 `user` 表，并插入一条基础数据。示例结构如下。

```sql
CREATE DATABASE users;
USE users;
CREATE TABLE user (
  id INT PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(64),
  password VARCHAR(64)
);
INSERT INTO user(username, password) VALUES ('admin', '123456');
```

### 3.4 应用配置设计

本题的关键不是把密码写错，而是让“代码写法正确、运行结果错误”。因此数据库配置建议写成：

```php
return [
    'type'            => 'mysql',
    'hostname'        => '127.0.0.1',
    'database'        => 'users',
    'username'        => 'root',
    'password'        => getenv('DATABASE_PASSWORD'),
    'hostport'        => '3306',
    'charset'         => 'utf8',
    'prefix'          => '',
];
```

同时在容器启动时设置：

```bash
export DATABASE_PASSWORD='Passw0rd@123456789'
```

这样，题面“数据库密码位于环境变量中”在事实层面成立，但由于 PHP-FPM 默认不向请求进程传递环境变量，业务访问依旧会失败。

## 四、故障触发机理设计

本题最核心的训练点在于 **CLI 环境与 Web/FPM 环境的差异**。因此制作环境时，必须保证以下现象同时成立。

| 验证项 | 预期现象 |
| --- | --- |
| `env | grep DATABASE_PASSWORD` | 能看到真实密码 |
| `php -r 'var_dump(getenv("DATABASE_PASSWORD"));'` | CLI 中可取到密码 |
| 浏览器访问临时 `envtest.php` | 返回 `bool(false)` |
| `getuser()` 或 `testmysql()` | 因数据库连接失败而报错 |

要实现这一点，最直接的方式就是保留 PHP-FPM 默认清理环境变量的行为，不在 `www.conf` 中开启如下配置：

```ini
clear_env = no
env[DATABASE_PASSWORD] = $DATABASE_PASSWORD
```

也就是说，**题目初始状态必须是这两项没有生效**，否则应用一开始就能连库，题目将失去训练意义。

## 五、判题逻辑设计

本题建议采用 **接口可用性判定型** 裁判，而不是命令回显型。裁判脚本不关心选手具体改了哪个文件，而只关心系统是否已经恢复到业务可用状态。

推荐的判题条件如下。

| 检查项 | 判定标准 |
| --- | --- |
| `Check.php::testmysql()` | 能成功返回指定 MD5 |
| 数据库查询 | 能正常从 `user` 表取到记录 |
| `/flag` | 条件满足后由裁判自动生成 |

示意性判题脚本如下：

```bash
#!/bin/sh
while true; do
  body=$(curl -sS http://127.0.0.1/public/index.php/index/check/testmysql 2>/dev/null)
  if [ "$body" = "8e74933d7c6fa65e1aab5769743447ed" ]; then
    echo 'xctf{REPLACE_WITH_REAL_FLAG}' > /flag
    chmod 644 /flag
    break
  fi
  sleep 5
done
```

这种设计的好处在于，既允许选手采用“规范修复”（修改 PHP-FPM 透传环境变量），也允许在权限受限的场景下采用“应急修复”（把真实密码临时写回配置），只要最终业务恢复即可通过。

## 六、标准解法与非标准解法

从复现设计角度看，本题最好允许两类修复路径同时成立。

| 修复路径 | 说明 | 是否建议作为官方预期 |
| --- | --- | --- |
| 规范修复 | 修改 PHP-FPM 池配置，使 Web 进程可读取 `DATABASE_PASSWORD` | 是 |
| 应急修复 | 直接把真实数据库密码写回 `database.php` | 可作为权限受限场景下的容错路径 |

之所以建议保留第二条路径，是因为这更贴近真实应急响应：当处置人员没有权限调整服务级配置时，往往会先采取最小变更恢复业务，再在后续窗口中完成彻底整改。

## 七、启动逻辑设计

建议使用统一启动脚本 `start.sh` 完成数据库初始化、环境变量设置、Web 服务启动、判题脚本拉起和 ttyd 暴露。推荐职责如下。

| 启动项 | 作用 |
| --- | --- |
| 初始化 MySQL 数据库与样例表 | 提供业务查询对象 |
| 设置 `DATABASE_PASSWORD` 环境变量 | 构造题面前提 |
| 启动 PHP-FPM 与 Nginx | 提供业务服务 |
| 启动判题脚本 | 持续检测修复状态 |
| 创建 `ctf` 用户并启动 ttyd | 提供在线终端 |

一个示意性 `start.sh` 可以写成：

```bash
#!/bin/bash
set -e

export DATABASE_PASSWORD='Passw0rd@123456789'

service mysql start
mysql -uroot -p"$DATABASE_PASSWORD" <<'SQL'
CREATE DATABASE IF NOT EXISTS users;
USE users;
CREATE TABLE IF NOT EXISTS user (
  id INT PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(64),
  password VARCHAR(64)
);
DELETE FROM user;
INSERT INTO user(username, password) VALUES ('admin', '123456');
SQL

php-fpm &
nginx &
/root/check/judge.sh &
/usr/bin/ttyd -p 18080 login
```

## 九、落地复现时的注意事项

本题在制作时有三点尤其需要注意。第一，必须保证环境变量在 **系统中存在**，否则题面本身就失真。第二，必须保证 Web/FPM 请求上下文默认**读不到**该变量，否则选手无法观察到“明明有环境变量却仍然连库失败”的核心现象。第三，判题逻辑应尽量围绕业务恢复本身，而不是强依赖某个特定修复动作，这样题目会更稳健，也更符合真实应急场景。

归根结底，这道题最值得保留的并不是某个 ThinkPHP 文件本身，而是 **题面提示、应用配置、FPM 环境清洗、业务校验接口、异步放旗** 共同组成的闭环。只要这个闭环被正确复刻，这道题就能很好地训练选手对 Web 服务配置故障的定位与修复能力。
