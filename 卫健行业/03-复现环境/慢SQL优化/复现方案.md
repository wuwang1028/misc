# 慢SQL优化题复现方案

## 一、复现目标

本方案用于后续在本地或私有靶场中复现“慢SQL优化”题目的运行环境、慢查询现象与判题逻辑。按照当前要求，本文档**仅描述复现方案，不直接落地完整容器或脚本实现**。目标是保证后续需要制作 Docker 环境时，可以直接依据本文档补齐镜像、初始化 SQL、判题脚本与访问入口。

## 二、环境设计目标

本题的核心体验由四个部分构成：一是 Web 终端访问入口；二是可登录的 Linux 用户环境；三是包含特定数据分布的 MariaDB 数据库；四是后台定期判题并在达标后生成 `/flag` 的机制。因此复现环境至少应保留如下能力。

| 模块 | 作用 | 复现要求 |
| --- | --- | --- |
| `ttyd` 终端 | 为选手提供浏览器内命令行交互 | 暴露一个 HTTP 入口，登录后进入系统 shell |
| Alpine Linux 容器 | 作为题目基础运行环境 | 保持轻量、命令集接近原题 |
| MariaDB 服务 | 承载 `test` 数据库与目标 SQL | 预置大表数据、原始索引状态与 root/root 凭据 |
| 判题脚本 | 周期性检测 SQL 是否优化到阈值内 | 达标后生成 `/flag` |
| 题目说明文件 | 指导选手完成操作 | 在 `/home/ctf/README.md` 写入提示 |

## 三、建议目录结构

后续真正落地复现环境时，建议采用如下结构，以便把数据库初始化、容器编排与判题逻辑分离管理。

| 路径 | 建议内容 |
| --- | --- |
| `Dockerfile` | 安装 Alpine 基础依赖、MariaDB、ttyd、bash/ash 工具 |
| `docker-compose.yml` | 编排题目服务，对外暴露 HTTP 端口 |
| `init/01-schema.sql` | 创建 `test` 数据库、`event_q` 和 `staff_event` 表结构 |
| `init/02-data.sql` | 导入题目规模的数据集 |
| `init/03-readme.sh` | 写入 `/home/ctf/README.md` 与 flag 占位逻辑 |
| `checker/judge.py` | 周期性执行目标 SQL，测量耗时，达标后生成 `/flag` |
| `checker/start.sh` | 启动 MariaDB、ttyd 与判题脚本 |
| `docs/题目说明.md` | 对外发布时使用的题面与运维说明 |

## 四、数据库复现要点

本题是否成立，关键取决于数据库中的**数据分布**而不只是表结构。仅仅创建两张空表并插入少量数据，无法复现“SQL 明显变慢但仍可通过索引和统计信息优化”的特征。因此在后续制作环境时，必须尽量还原如下属性。

### 1. 表结构

建议复现与原题一致的基础索引状态，即：

- `event_q` 仅保留 `PRIMARY(event_id)`；
- `staff_event` 仅保留 `PRIMARY(event_id, staff_id)`；
- 不预先建立 `idx_event_type_date_id` 与 `idx_staff_read_event`；
- 不提前启用持久统计修正动作，让选手自己完成。

### 2. 数据规模

根据当前实测，原题中至少具有如下数量级：

| 表名 | 目标数量级 |
| --- | ---: |
| `event_q` | 20 万级 |
| `staff_event` | 80 万级 |

### 3. 数据分布

为了让题目具备可玩性，数据生成脚本至少应满足以下特征：

- `event_q.event_type='2'` 且日期位于 `2023-01-01` 到 `2023-08-12` 的记录量较大，接近十万级；
- `staff_event` 中 `staff_id=1 and is_read=1` 的结果集明显更小，约千级；
- 两表连接后命中约数百行，并且这些记录中的 `content` 字段包含关键词“权限”；
- `content like '%权限%'` 不能被普通 B-Tree 索引直接优化，使选手必须依赖连接顺序与统计信息。

如果后续需要自行造数，建议先固定命中集，再大量填充噪声数据，以确保：**从 `event_q` 驱动会慢，从 `staff_event` 驱动会快**。

## 五、判题逻辑设计

判题脚本建议独立于选手终端运行，并以周期轮询方式执行目标 SQL。其核心判断条件不应是“检测某个索引名是否存在”，而应是**真实执行耗时**是否低于阈值。这样更符合题目“慢SQL优化”的本意，也能避免选手通过无效索引命名投机过关。

建议判题逻辑如下：

| 步骤 | 建议实现 |
| --- | --- |
| 连接数据库 | 使用 `root/root` 连接 MariaDB |
| 执行查询 | 直接运行题面给定 SQL |
| 测量耗时 | Python 中使用 `time.perf_counter()` 包裹执行过程 |
| 通过阈值 | 建议设为 `0.1s` 或 `0.2s` 以内，可按实际环境调优 |
| 结果处理 | 若达标且 `/flag` 不存在，则写入 `/flag` |
| 轮询频率 | 每 5 到 10 秒一次 |

这里建议将 flag 生成为一个普通文件，例如 `/flag`，并赋予可读权限，以便选手在优化成功后能立即通过 `cat /flag` 获取结果。

## 六、访问与账号设计

题目体验上应尽量保持与原题一致，即让选手通过浏览器进入 `ttyd` 页面后，使用普通账号登录系统，再手动操作数据库。建议设计如下。

| 项目 | 建议值 |
| --- | --- |
| 系统账号 | `ctf` |
| 系统密码 | `ctf` |
| 数据库账号 | `root` |
| 数据库密码 | `root` |
| 终端命令 | `login` 或直接进入 `/bin/sh` |
| README 路径 | `/home/ctf/README.md` |

`README.md` 中应明确说明：题目目标、给定 SQL、数据库账号密码、禁止修改 root 密码、优化完成后自动生成 `/flag`。这样既保留题目引导，又不至于泄露解法。

## 七、后续实现建议

当后续正式开始落地复现环境时，建议按以下顺序实施：首先编写 SQL 初始化文件，把表结构与样本数据固定下来；随后在本地手动验证“原始查询确实慢、加索引后仍不够、启用持久统计后明显变快”；再将这组数据库状态打包进 MariaDB 初始化流程；最后补齐 `ttyd` 与判题脚本，使整体环境可以一键启动。

从当前已解出的原题结果看，这道题最重要的教学点不是“记住一条命令”，而是让选手理解：**慢查询优化依赖于索引设计、数据分布和优化器统计三者共同作用**。因此在复现环境设计时，务必把这个核心体验保留下来，而不是把题目做成纯粹的“建索引签到题”。

## 八、当前已验证的最终优化动作

为了便于后续复现时对照，本题在原环境中最终有效的优化动作如下：

```sql
use test;
create index idx_event_type_date_id on event_q(event_type, create_date, event_id);
create index idx_staff_read_event on staff_event(staff_id, is_read, event_id);
analyze table event_q, staff_event;
set global use_stat_tables='PREFERABLY';
analyze table event_q persistent for all;
analyze table staff_event persistent for all;
```

这组操作最终将目标 SQL 的执行时间从约 `5.35s` 压缩到约 `0.024s`，并成功触发 `/flag` 生成。后续在复现环境验收阶段，可以将其作为基准答案进行回归测试。
